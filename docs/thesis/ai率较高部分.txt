平台采用前后端分离架构：前端以 Vue 3 配合 Element Plus 构建响应式界面，后端基于 Spring Boot 3.2 提供 RESTful API，数据层由 MySQL 8.0 承担事务存储、Redis 负责缓存加速。业务层面，平台打通了从商品发布、订单流转到支付宝沙箱支付的全链路交易闭环，并以 WebSocket 协议支撑买卖双方的实时通讯。AI 侧的整合是 SwapU 区别于一般交易系统的核心。Spring AI 框架统一了 LLM 调用入口，Elasticsearch 接管了全文检索。在导购环节，平台用 RAG 技术搭配 PgVector 向量数据库，构建了具备上下文理解能力的智能导购助手——用户以自然语言提问，系统先检索真实在售商品再交由模型生成推荐，从而抑制了大模型的幻觉问题。在发布环节，大语言模型通过反复调优的 System Prompt 生成结构化商品描述文案，显著降低了信息呈现门槛。测试数据表明，SwapU 在 500 线程持续施压 5 分钟的条件下，核心交易链路没有出现状态断点或脏数据。RocketMQ 延时队列保障了订单超时场景下的状态一致性与商品状态回滚。整体系统将多组件协同架构与 AI 技术融合为完整可用的工程方案，为大型语言模型在垂直电商领域的落地提供了实证参考。



SwapU 的设计正是围绕上述三个支点展开的。平台基于 Spring Boot 3.2 与 Vue 3 构建了完整的交易链路，更关键的是将 Spring AI 框架与大语言模型嵌入到了两个核心业务环节。在发布侧，卖家只需输入关键信息并点击 AI 一键润色，系统通过反复调优的 SystemPrompt 调用大模型生成结构化商品描述文案。在导购侧，用户以自然语言对话即可获得基于真实在售商品的智能推荐。

二手交易在商业形态上已形成了较为成熟的范式。国外方面， eBay 率先将分类信息数字化并构建了基于信誉评分的交易体系， Craigslist 以极简信息发布降低用户参与门槛，Poshmark 则融入了社交互动与直播元素以提升平台黏性。国内方面，闲鱼依托支付宝的芝麻信用体系构建了 C2C 交易信任机制，转转在手机等 3C 品类上凭借质检和官方验机服务实现了差异化定位。上述平台的共同范式是 C2C 或 C2B2C 模式下的全国性撮合，在校园封闭场景中的适用性有限。从技术演进的角度看，二手电商平台的搜索能力经历过几个阶段的迭代。最早采用数据库 LIKE 模糊查询，随后基于 Elasticsearch 的 BM25 算法配合协同过滤技术显著提升了关键词匹配的精度和召回率，但对于口语化、带有隐含意图的查询，语义理解层面仍存在明显短板。大语言模型的突破为解决语义理解的困境提供了新的技术路径。RAG 的核心思想是先检索后生成[15]：在将用户查询交给大模型之前，先从外部知识库中检索出最相关的真实数据，再将这些数据作为参考上下文注入 Prompt[16 - 17]，使模型的回答被约束在真实数据范围内。在垂直领域的知识服务中，已有研究构建了结合知识库与知识图谱的 RAG 问答系统[5]。针对电商智能客服场景，相关研究将 RAG 与多语言处理结合[6]，验证了检索增强在垂直电商中的可行性。然而，从学术验证到生产级部署， RAG 系统面临着一系列工程层面的挑战。商品库处于持续变动之中——上新、下架、改价、成交——如何保证向量数据库中的 Embedding 与关系型数据库中的行记录保持同步？消息丢失时如何处理？这些问题在学术论文中通常不展开讨论，但在实际部署中每一个都可能导致系统向用户输出错误推荐。本课题的 SwapU 系统将注意力集中在这些工程落地的关键问题上。



本系统的技术选型基于解决具体工程痛点的原则。选用 Vue 3 主要基于以下考量：Composition API 在处理商品发布等包含多状态联动的复杂组件时，能够将关联逻辑高度聚合，从而显著降低维护成本。后端选用 Spring Boot 3 . 2 结合 MyBatis - Plus ，主要旨在利用其条件构造器简化复杂动态 SQL 的编写。以下按前端、后端、中间件、 AI 四个板块展开。


随着 Web 技术的演进，传统的单体架构逐渐暴露出扩展性差、前后端耦合度高等瓶颈。大量研究与工程实践表明，采用前后端分离架构与微服务模式能够显著提升系统开发的灵活性与可维护性[7 - 9]。Vue 3 引入了 Composition API 和基于 Proxy 的响应式系统重构， Composition API 的核心优势在于逻辑复用——以函数为粒度组织代码，使相关逻辑聚合。Vite 利用浏览器原生 ESM 支持，开发服务器启动速度达到毫秒级。Element Plus 提供 60 余种开箱即用的企业级组件。Pinia 替代 Vuex 作为状态管理方案， API 更简洁且原生支持 TypeScript。Axios 拦截器统一处理 JWT 令牌注入和 401 响应跳转。


在处理高并发下单与数据一致性问题时，合理的中间件应用至关重要。Redis 在缓存加速、布隆过滤器防穿透以及分布式锁方面的应用价值已被广泛论证[11]，这为本系统采用 Redisson 实现防超卖机制提供了工程参考。Redis 在系统中承担了缓存、分布式锁和限流三项职责，三者共享同一个 RedissonClient 实例。RocketMQ 管理两个 Topic。item - update - topic 负责在商品发布、修改或状态变更后，将 ItemSyncMessage 异步推送至 ItemSyncListener ，由消费者完成 Elasticsearch 索引和 PgVector 向量的同步写入或删除。order - timeout - topic 负责订单超时自动取消——下单时发送一条延时 30 分钟的消息，到期后消费者检查订单状态并执行取消逻辑。Elasticsearch 的职责定位清晰：MySQL 负责写操作与事务性存储， ES 负责搜索读操作。采用 MySQL 的 LIKE 语句进行模糊搜索存在明显的性能瓶颈， ES 的倒排索引结构配合 IK分词插件在全文检索场景下的查询效率远优于关系型数据库。PostgreSQL 16 配合 pgvector 扩展，提供向量存储和相似度检索能力，支持 HNSW 索引加速余弦相似度计算


AI 层面的技术选型遵循一个核心前提：后端不应与某一家大模型厂商强绑定。Spring AI 的 ChatClient 和 EmbeddingClient 两个核心抽象正是为此设计——底层模型供应商的切换对上层业务代码透明。RAG 管线在工程层面由三个物理组件构成：PgVector 负责存储商品 Embedding 向量，索引类型选用 HNSW ，距离度量采用余弦距离；Ollama 本地部署 bge - m3 模型提供 Embedding 服务；DeepSeek Chat 以 OpenAI 兼容格式提供对话能力。向量嵌入是将文本转化为固定维度浮点数向量的技术，语义相近的文本在向量空间中距离更近。bge - m3 是北京智源人工智能研究院发布的开源 Embedding 模型，支持多语言和多粒度嵌入，输出维度为 1024。PgVector 是 PostgreSQL 的向量扩展，支持 HNSW 索引加速近似最近邻搜索。DeepSeek Chat 在用户发起导购对话时，基于系统注入的检索结果进行引导式推荐。这条链路中最容易被忽视的环节不是模型选型本身，而是 System Prompt 的调优。文案润色功能经历了十余个版本的迭代，最终版本的 Prompt 明确规定了字数、格式和内容约束， temperature 参数设为 0 . 1 以降低随机性。智能导购功能的 Prompt 更为复杂。除了角色定位和输出格式的约束外，必须注入严禁编造不存在的商品这一强制指令，以及从 PgVector 中实时检索出的 Top 3 商品元数据。该元数据是 RAG 管线的核心约束手段——将模型输出锚定在真实商品数据上，从源头抑制幻觉。



智能导购功能的 Prompt 更为复杂。除了角色定位和输出格式的约束外，必须注入严禁编造不存在的商品这一强制指令，以及从 PgVector 中实时检索出的 Top 3 商品元数据。该元数据是 RAG 管线的核心约束手段——将模型输出锚定在真实商品数据上，从源头抑制幻觉。

技术层面的评估基于现有开源生态的成熟度。全栈的每一个关键组件均基于社区活跃的开源方案，技术风险可控。经济可行性方面，开发工具和所有框架、中间件均为开源且免费使用。操作可行性方面，目标用户群体对电商类应用和即时通讯工具的交互模式已高度熟悉，无需额外学习成本。
经济可行性的评估较为直接。开发工具方面， IDE 社区版及所有框架、中间件均为开源且免费使用。部署方面，各中间件服务及 Ollama 本地推理均可运行在单台 4 核 8 GB 的虚拟机上，硬件成本在学生可承受范围内。
操作可行性方面，目标用户群体（在校大学生）对电商类应用和即时通讯工具的交互模式已高度熟悉。Element Plus 的卡片式信息流、底部导航栏、聊天界面等组件与主流移动应用的交互语言一致，用户无需额外的学习成本即可完成基本操作。

在检索与导购流程中，买家获取商品有两条路径。路径一为关键词检索：用户在搜索框输入查询词， Elasticsearch 执行基于 BoolQuery 的复合查询返回结果。路径二为 AI 导购：用户以自然语言描述需求，系统通过 RAG 管线检索真实在售商品后交由大模型生成推荐。


（ 1 ）用户认证与个人中心模块。提供基于 BCrypt 加密的注册与登录功能，登录成功后签发 JWT Token。个人中心支持查看已发布商品、收藏商品列表、买入/卖出订单的查询与管理。 （ 2 ）商品浏览与检索模块。首页以卡片式信息流展示在售商品，顶部按分类层级提供导航。检索功能基于 Elasticsearch 的 BoolQuery 复合查询，支持关键词搜索、分类筛选和高亮标注。（ 3 ） AI 智能交互模块。包含发布阶段的一键文案润色功能和独立的 AI 智能导购对话界面。导购助手需具备两项核心能力：理解用户的自然语言购物意图并给出针对性推荐；支持多轮对话上下文记忆。 （ 4 ）即时通讯模块。买卖双方通过 WebSocket 建立实时私信通道，支持文本和图片消息，消息持久化至数据库。 （ 5 ）订单与支付流转模块。覆盖下单、支付拉起、状态查询、交易取消与确认收货的完整链路。系统需在高并发场景下保证同一商品不会被多个买家同时下单，并限制单用户的下单频率。（ 1 ）数据大盘模块。以 ECharts 图表展示平台关键运营指标——近 7 日订单量趋势、新增用户数、商品品类分布以及大模型接口调用频次。 （ 2 ）内容审核模块。管理员可对全平台商品、留言及用户举报进行统一管理，支持对违规商品执行强制下架操作。举报处理结果联动更新被举报对象的状态字段，确保审核操作形成闭环。

非功能性需求中，以下几个方面在本系统中具有刚性约束： 第一，响应时间与限流。首页商品列表接口需在 200 毫秒内返回结果，搜索接口需在 500 毫秒内返回。下单接口因涉及并发控制与数据库事务，可放宽至 500 毫秒。限流方面，下单接口需限制单用户每分钟最多 3 次请求，防止恶意刷单；登录接口当前未配置限流，生产环境需补充。
第二，数据安全。用户手机号等敏感字段不得以明文形式写入数据库，需在数据层实现透明加解密，确保即使数据库被直接访问也无法获取用户隐私信息。所有涉及资源归属的操作需校验当前用户身份，防止越权访问。 第三，系统可维护性。AI 模块与业务模块之间需通过接口约定进行解耦——底层对话模型和向量库的具体选型对上层调用方应保持透明，以便后续独立替换。日志方面，系统需在 Controller 层实现全量审计日志。
