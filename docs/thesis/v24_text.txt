P65: 摘    要
P66: 高校闲置物品的周期性积压是一个普遍现象，每学期末大量教材、电子设备等物资因学业阶段切换而被淘汰，而新生群体年复一年存在刚性采购需求，二者之间存在结构性错配。现有主流二手交易平台（如闲鱼、转转）以全国大市场为架构基因，在校园场景下存在信息匹配效率不足、交易摩擦过大、发布门槛偏高三方面错位。与此同时，大语言模型与检索增强生成（RAG）技术的快速发展，为解决上述痛点提供了新的技术路径。
P67: 针对上述问题，本文设计并实现了一个基于Vue3+SpringBoot 的校园闲置物品循环利用系统——SwapU。系统采用前后端分离架构，前端基于 Vue 3 + Element Plus + Pinia 构建，后端基于 Spring Boot 3.2 + MyBatis-Plus 开发，数据层整合 MySQL、Redis、Elasticsearch 和 PgVector 四种存储引擎。系统的核心创新在于将大语言模型深度整合进业务流程：发布侧通过 DeepSeek Chat 实现一键文案润色，降低卖家发布门槛；导购侧基于 RAG 管线，先从 PgVector 检索平台真实在售商品，再将检索结果注入上下文交由模型生成推荐，从机制层面抑制幻觉输出。此外，系统通过 WebSocket 实现买卖双方即时通讯，通过支付宝沙箱完成支付闭环，通过 RocketMQ 延时消息实现订单超时自动取消。
P68: 在工程实现层面，本文重点解决了三个关键问题：第一，通过 RocketMQ 消息驱动 + 全量同步的双轨策略保证 PgVector 向量数据与 MySQL 行记录的一致性，确保 AI 导购推荐的商品均为真实在售状态；第二，通过十余版本的 System Prompt 迭代调优，将 AI 输出行为收敛至稳定可控状态，解决大模型输出格式不统一、字数超标和编造商品信息等问题；第三，通过 Redisson 分布式锁与编程式事务的配合，有效防止并发下单场景下的商品超卖。系统测试结果表明，各功能模块运行正常，AI 导购推荐准确率满足预期，AES 加密有效保护用户隐私，系统整体运行稳定。
P69: 关键词：校园二手交易；检索增强生成；大语言模型；Spring Boot；Vue 3
P70: Abstract
P71: The periodic accumulation of idle items in universities is a common phenomenon. At the end of each semester, a large number of textbooks, electronic devices, and other materials are discarded due to academic stage transitions, while new students consistently have rigid purchasing demands year after year, creating a structural mismatch between supply and demand. Existing mainstream second-hand trading platforms (such as Xianyu and Zhuanzhuan) are architecturally designed for a national market, resulting in three mismatches in campus scenarios: insufficient information matching efficiency, excessive transaction friction, and high publishing barriers.
P72: To address these issues, this paper designs and implements SwapU, a campus intelligent second-hand trading system based on Retrieval-Augmented Generation (RAG). The system adopts a front-end and back-end separation architecture, with the front-end built on Vue 3 + Element Plus + Pinia and the back-end developed using Spring Boot 3.2 + MyBatis-Plus. The data layer integrates four storage engines: MySQL, Redis, Elasticsearch, and PgVector. The core innovation of the system lies in the deep integration of large language models into business processes: on the publishing side, DeepSeek Chat enables one-click copywriting polish to lower the seller's publishing barrier; on the shopping guide side, the RAG pipeline first retrieves real listed products from PgVector, then injects the retrieval results into the context for the model to generate recommendations, suppressing hallucinated output at the mechanism level.
P73: At the engineering implementation level, this paper focuses on solving three key problems: first, maintaining consistency between PgVector vector data and MySQL row records through a dual-track strategy of RocketMQ message-driven incremental sync and full sync, ensuring that AI-recommended products are genuinely available; second, converging AI output behavior to a stable and controllable state through more than ten iterations of System Prompt tuning, resolving issues such as inconsistent output formats, excessive word counts, and fabricated product information; third, preventing overselling in concurrent order scenarios through the combination of Redisson distributed locks and programmatic transactions. System testing results show that all functional modules operate normally, AI shopping guide recommendation accuracy meets expectations, AES encryption effectively protects user privacy, and the system runs stably overall.
P74: Key Words: Campus Second-hand Trading; Retrieval-Augmented Generation; Large Language Model; Spring Boot; Vue 3
P76: 第1章 绪论
P77: 1.1 课题研究背景及意义
P78: 高校闲置物品的周期性积压，并不是一个新现象。每学期末，宿舍楼下成摞的考研书、没拆封的生活用品、用了几个月的电子设备——这批物资被淘汰的原因并非丧失了使用价值，而是原主人的学业阶段发生了切换。一面是巨量闲置品的持续产生，另一面是新生群体年复一年的刚性采购需求。二者的错配是结构性的。与此同时，二手交易中买卖双方的信任缺失进一步阻碍了交易的达成[1]。因此，构建一个高效便捷的校园二手交易生态，在资源循环利用和绿色消费两个维度上都具有切实的现实意义。
P79: 闲鱼和转转为什么没能有效覆盖这一需求？它们的架构基因是全国大市场——算法优先推荐跨区域优质卖家，物流体系围绕快递网络设计。这套逻辑放在校园场景里出现了三方面错位：信息匹配效率不足，搜索者关心的是对方是否在相邻宿舍楼而非全国信用分；交易摩擦过大，校园场景以当面交易为主但平台缺乏同城面交支持；发布门槛偏高，学生不愿为短期闲置品撰写详细描述。此外，现有平台的全国大市场模式难以适配校园场景的本地化需求[2]，而社区化的闲置物品处置模式已被研究证实能有效提升资源流转效率[3]。校园交易具有明显的周期性特征：每学期末是闲置品释放高峰，每学期初是采购需求高峰，而现有平台的推荐算法并未针对这种周期性进行优化。
P80: SwapU 的设计围绕上述三个痛点展开。底层交易链路用 Spring Boot 3.2 与 Vue 3 搭建，属于常规开发工作。真正需要反复调试的是 AI 与业务的整合，最终落地在两个环节：发布侧，卖家输入关键词后点击 AI 润色，后端调用 DeepSeek 生成结构化商品描述——该功能的 System Prompt 经过十余个版本的迭代才收敛至稳定输出，使模型输出行为符合预期指令[19]；导购侧，用户以自然语言提问，系统先从 PgVector 中检索真实在售商品，再将检索结果注入上下文交由模型生成推荐，从源头避免模型编造不存在的商品信息。
P81: 1.2 国内外研究现状
P82: 二手交易在商业形态上已形成了较为成熟的范式。国外方面，eBay 率先将分类信息数字化并构建了基于信誉评分的交易体系，其拍卖模式至今仍是 C2C 交易的标杆；Craigslist 以极简信息发布降低用户参与门槛，证明了轻量级信息撮合在本地化交易中的有效性；Poshmark 则融入了社交互动与直播元素以提升平台黏性，开创了社交电商的先河。国内方面，闲鱼依托支付宝的芝麻信用体系构建了 C2C 交易信任机制，通过社区化运营提升用户活跃度；转转在手机等 3C 品类上凭借质检和官方验机服务实现了差异化定位，解决了高价值二手商品信任缺失的痛点。上述平台的共同范式是 C2C 或 C2B2C 的大市场撮合模式，对校园场景的地理聚合和即时交付特性缺乏针对性设计，在交互体验层面亦未能针对学生群体的使用习惯进行优化[4]。在学术研究层面，已有基于 Spring Boot 的校园二手交易系统实现[14]，但未引入 AI 能力，商品检索仍依赖传统关键词匹配，无法理解用户的语义化购物意图。
P83: 从技术演进的角度看，二手电商平台的搜索能力经历过几个阶段的迭代。最早采用数据库 LIKE 模糊查询，性能和精度均存在明显瓶颈；随后基于 Elasticsearch 的 BM25 算法配合协同过滤技术显著提升了关键词匹配的精度和召回率，但对于口语化、带有隐含意图的查询，语义理解层面仍存在明显短板。例如用户输入“有没有考研资料”，传统搜索引擎只能匹配包含“考研”关键词的商品标题，无法理解用户实际需要的是考研相关教材和真题。大语言模型的突破为解决语义理解的困境提供了新的技术路径[18]，但直接使用大模型存在幻觉问题——模型可能编造不存在的商品推荐，这在电商场景中是不可接受的。
P84: RAG 的核心思想是先检索后生成[15]：在将用户查询交给大模型之前，先从外部知识库中检索出最相关的真实数据，再将这些数据作为参考上下文注入 Prompt[16-17]，使模型的回答被约束在真实数据范围内。RAG 机制从根本上缓解了大模型的幻觉问题，使生成内容具备可追溯的事实依据。在垂直领域的知识服务中，已有研究构建了结合知识库与知识图谱的 RAG 问答系统[5]，验证了 RAG 在专业领域中的有效性。针对电商智能客服场景，相关研究将 RAG 与多语言处理结合[6]，验证了检索增强在垂直电商中的可行性。这些研究为 SwapU 的 AI 导购功能设计提供了理论支撑和技术参考。
P85: 然而，从学术验证到生产级部署，RAG 系统面临着一系列工程层面的挑战。商品库处于持续变动之中——上新、下架、改价、成交——如何保证向量数据库中的 Embedding 与关系型数据库中的行记录保持同步？消息丢失时如何处理？这些问题在学术论文中通常不展开讨论，但在实际部署中每一个都可能导致系统向用户输出错误推荐。本课题的 SwapU 系统将注意力集中在这些工程落地的关键问题上。
P86: 本课题的 SwapU 系统正是在上述背景下提出的。与已有研究不同，SwapU 不满足于在实验室环境中验证 RAG 的可行性，而是将其部署到真实的校园交易场景中，面对商品库持续变动、用户查询口语化、模型输出不可控等实际工程挑战，逐一给出解决方案。SwapU 的设计目标是在保证基础交易功能完整的前提下，通过 AI 能力降低发布门槛和提升导购效率，使校园二手交易从信息撮合模式升级为智能服务模式。
P87: 1.3 主要研究内容
P88: 本课题围绕&quot;如何将 RAG 技术有效整合进校园二手交易场景&quot;这一核心问题展开，研究内容分为三个层次：
P89: （1）基础交易链路的构建。基于 Spring Boot 3.2 与 Vue 3 搭建前后端分离的校园闲置物品循环利用平台，覆盖商品发布、全文检索、即时通讯、订单交易与支付闭环等基础功能，确保从发布到交易的业务链路完整可用。
P90: （2）AI 与业务的深度整合。发布侧研究如何通过 System Prompt 迭代调优使大模型输出收敛至稳定可控状态，降低卖家文案撰写门槛；导购侧研究如何基于 RAG 管线实现&quot;先检索后生成&quot;的推荐机制，从源头抑制模型幻觉，使推荐结果基于平台真实在售商品。
P91: （3）工程落地的关键问题。重点解决向量数据与关系数据的一致性保障（双轨同步策略）、并发下单的防超卖控制、敏感字段的零侵入加密等工程挑战，确保系统在真实部署环境下的正确性与可靠性。

P102: 2.1 功能需求分析
P103: SwapU 校园闲置物品循环利用系统面向在校大学生，旨在提供一个便捷高效的校园二手物品交易平台，使大学生能够更轻松地发布、检索和交易闲置物品。系统利用 Spring Boot 框架、Elasticsearch 搜索引擎和 RAG 检索增强生成技术，为用户提供智能化的交易体验。在功能设计上，系统遵循“发布-检索-交易-沟通”的核心链路，确保每个环节都有对应的功能支撑。
P104: 本系统的用户角色通过 User 实体中的 role 字段区分，分为两类：普通用户（role=0，对应买卖双方）和系统管理员（role=1）。未登录用户通过 LoginInterceptor 的白名单机制仅可访问登录、注册和商品浏览页面。普通用户可执行商品发布、搜索、购买、即时通讯和 AI 交互等全部前台功能；管理员除拥有普通用户权限外，还可访问后台管理页面，执行商品审核、举报处理和公告发布等管理操作。
P105: 在上述功能需求的基础上，前台交易子系统的核心功能模块如下：
P106: （1）用户认证与个人中心模块。提供基于 BCrypt 加密的注册与登录功能，登录成功后签发 JWT Token。个人中心支持查看已发布商品、收藏商品列表、买入/卖出订单的查询与管理。
P107: （2）商品浏览与检索模块。首页以卡片式信息流展示在售商品，顶部按分类层级提供导航。检索功能基于 Elasticsearch 的 BoolQuery 复合查询，支持关键词搜索、分类筛选和高亮标注。
P108: （3）AI 智能交互模块。包含发布阶段的一键文案润色功能和独立的 AI 智能导购对话界面。导购助手需具备两项核心能力：理解用户的自然语言购物意图并给出针对性推荐；支持多轮对话上下文记忆。
P109: （4）即时通讯模块。买卖双方通过 WebSocket 建立实时私信通道，支持文本和图片消息，消息持久化至数据库。
P110: （5）订单与支付流转模块。覆盖下单、支付拉起、状态查询、交易取消与确认收货的完整链路。系统需保证同一商品不会被多个买家同时下单，下单接口需通过分布式锁对商品加锁，防止并发场景下的超卖问题。
P111: 后台管理子系统的核心功能侧重于数据监控与平台治理：
P112: （1）数据大盘模块。以 ECharts 图表展示平台关键运营指标——近 7 日订单量趋势、新增用户数、商品品类分布以及大模型接口调用频次。管理员可通过大盘快速了解平台运营状况，及时发现异常波动。

P114: 2.2 业务流程分析
P115: SwapU 的业务核心可归纳为三条主线，三者之间存在清晰的顺序依赖：商品发布生成可检索的商品池，买家通过搜索或 AI 导购发现商品后产生购买意向进而生成订单，订单流转过程中买卖双方通过即时通讯贯穿始终。以下分别描述三条主线的业务流程。
P116: 在商品发布流程中，卖家登录后进入发布页面，上传商品实物图片并选择分类。为降低文案撰写门槛，卖家可在输入框中填入简要关键词后点击 AI 一键润色按钮，前端将请求发送至 /ai/polish 接口，后端调用 DeepSeek Chat 生成结构化文案。文案生成后自动填充至描述输入框，卖家可在此基础上进行修改。确认后提交发布请求，后端先调用 AI 审核接口检测合规性，审核通过后商品进入在售状态，同时异步写入 Elasticsearch 索引和 PgVector 向量库，确保商品可被搜索和 AI 导购检索。商品发布业务流程图如图 2-1 所示。
P119: 图 2-1 商品发布业务流程图
P120: 买家检索商品可通过两条路径实现。一条是传统关键词搜索：用户输入查询词，Elasticsearch 通过 BoolQuery 匹配 title 和 description 字段，经 IK 分词后返回结果列表，匹配关键词以高亮标签标注。另一条是 AI 导购：用户以自然语言描述需求，例如“有没有考研资料”，系统先将查询语句转化为向量在 PgVector 中检索，获取真实在售商品后再由 DeepSeek 生成推荐。两条路径最终均指向商品结果，区别在于 AI 导购能够理解口语化的购物意图，而关键词搜索不具备语义理解能力。检索与导购业务流程图如图 2-2 所示。

P123: 在订单交易流程中，买卖双方通过 WebSocket 通道就商品细节进行沟通。沟通完成后买家确认购买，点击立即购买按钮，前端调用 createOrder 接口创建订单并将商品状态更新为交易中，同时发送 RocketMQ 延时消息用于超时自动取消。买家点击支付后，系统调用支付宝沙箱生成支付表单并跳转至支付宝收银台，买家完成支付后支付宝通过异步回调通知后端，后端执行 RSA 验签和幂等性检查后更新订单支付状态。交易完成后买家可对商品进行评价。若买家 30 分钟内未完成支付，RocketMQ 延时消息触发自动取消，商品恢复在售状态。

P126: 2.3 非功能需求分析
P127: 非功能性需求中，以下几个方面在本系统中具有刚性约束：
P128: （1）响应时间需求。首页商品列表接口需在 200 毫秒内返回结果，搜索接口需在 500 毫秒内返回。下单接口因涉及数据库事务提交和消息发送，可放宽至 500 毫秒。AI 接口因包含大模型推理，响应时间在 3-10 秒之间，属于可接受范围。
P129: （2）数据安全需求。手机号等敏感字段不应以明文形式存储，数据库层面的加密设计是保障数据安全的重要手段[12]。若数据库遭遇泄露，用户隐私将面临直接暴露风险。本系统在 MyBatis-Plus 的 TypeHandler 中嵌入 AES-128 加解密逻辑，写入时自动加密、读出时自动解密，业务代码无需感知加密过程。用户密码采用 BCrypt 哈希存储，即使数据库泄露也无法还原明文。此外，所有涉及资源归属的接口均须校验当前登录用户 ID，防止越权操作他人数据。管理员接口通过 AdminInterceptor 校验 role=1，普通用户无法访问后台管理功能。
P130: （3）系统可维护性需求。AI 模块与业务模块之间需通过接口约定进行解耦，底层对话模型和向量库的具体选型对上层调用方应保持透明，以便后续独立替换。日志方面，系统需在 Controller 层实现全量审计日志。

P162: 第3章 系统设计
P163: 3.1 系统结构设计
P164: SwapU 采用前后端分离的 B/S 架构。系统的运行时结构可拆分为五层，每一层的职责边界清晰、互不重叠。系统整体架构如图 3-1 所示。
P167: 图 3-1 系统分层架构图
P168: 表现层由 Vite 开发服务器提供代理：/api 前缀的动态请求通过 Vite 的 proxy 配置转发至后端 8080 端口，前后端从部署层面完全解耦。前端采用 Vue 3 的 Composition API 组织代码，Vue 的组件化架构和响应式数据绑定机制为复杂交互页面的开发提供了高效支撑[10]。状态管理使用 Pinia 替代 Vuex 以简化数据流，UI 组件库选用 Element Plus。Axios 拦截器统一处理 token 注入和 401 跳转，各业务模块无需重复编写鉴权逻辑。
P169: 网关控制层在请求进入业务代码前完成鉴权与审计。LoginInterceptor 从 Authorization 请求头获取 token，调用 JwtUtils.validateToken() 解析 Claims 中的 userId、username、role，存入 Request 域供后续 Controller 使用。未登录用户访问受保护接口时返回 401 状态码，前端拦截器自动跳转至登录页。AdminInterceptor 在 LoginInterceptor 之后执行，校验 role=1 方可访问 /admin/** 路径，否则返回 403。GlobalLogAspect 通过 AOP 切面记录方法耗时与异常，为线上问题排查提供审计日志。
P170: 业务逻辑层包含 15 个 Controller 和 13 个 ServiceImpl，基本一一对应。Controller 层仅负责参数校验和响应封装，业务逻辑全部下沉至 Service 层。ItemServiceImpl 依赖最密集，除 CRUD 外还承担 ES 同步和 RAG 向量写入。TradeOrderServiceImpl 封装订单创建、支付回调与状态流转逻辑，是系统中事务密度最高的模块。RagServiceImpl 封装向量检索逻辑，供 ChatController 调用。ChatMessageService 处理 WebSocket 消息的持久化与已读状态更新。
P171: 数据持久层各存储引擎职责隔离。MySQL 承担 ACID 写入和关联查询，是系统的主数据源。Redis 在缓存和分布式锁等场景中具有成熟的应用实践[11]。本系统中 Redis 承载缓存热点商品详情以降低 MySQL 查询压力，同时通过 Redisson 分布式锁为下单接口提供防超卖保障。Elasticsearch 承担全文检索，通过 RocketMQ 异步同步商品数据，查询效率相比 MySQL LIKE 提升数个数量级。PgVector 承担向量语义检索，商品上架时异步写入 Embedding 向量，为 RAG 管线提供事实数据源。
P172: 基础设施层包含四个外部系统。RocketMQ 维护 item-update-topic 和 order-timeout-topic 两个 Topic。支付宝沙箱提供 PC 网站支付，通过异步回调更新订单支付状态。Ollama 本地部署 bge-m3 模型提供 Embedding 服务。DeepSeek API 以 OpenAI 兼容格式提供 Chat 能力。

P196: 3.3 功能模块设计
P197: 系统在逻辑上划分为四个模块，模块间通过接口通信，禁止跨模块直接访问对方的数据库表。这种模块化设计使各模块可独立开发和测试，降低了系统耦合度。以下分别阐述各模块的职责和关键设计决策。
P198: 基础支撑模块管理所有业务共同依赖的横切逻辑，其安全机制的设计参考了轻量级安全Web开发框架的研究思路[13]。密码用 BCrypt 哈希存储，登录验证时通过 PasswordEncoder.matches() 比对，避免明文传输和存储。JWT 签发逻辑封装在 JwtUtils 中，密钥通过 Keys.hmacShaKeyFor() 从配置注入的 secret 派生，采用 HMAC-SHA256 算法，Claims 包含 userId、username、role 三个字段，过期时间 24 小时。JWT 的无状态特性使系统天然支持水平扩展，无需共享 Session 存储。手机号通过 AES-128 TypeHandler 加密存储，密钥配置在 application.yml 中，加解密过程对业务代码完全透明。
P199: 核心交易模块的代码量最大，涵盖商品、订单和支付三个子模块。商品子模块维护五态状态机：status=0 待审核、1 在售、2 交易中、3 已售出、4 已下架。状态只能由后端在特定业务操作中变更，前端无法直接修改状态字段。商品发布时异步写入 ES 索引和 PgVector 向量，确保可搜索和可导购。订单子模块同样维护四态状态机：0 待确认、1 进行中、2 已完成、3 已取消。订单创建时通过 RocketMQ 延时消息实现 30 分钟超时自动取消。支付子模块对接支付宝沙箱，异步回调中通过 RSA 验签和幂等性检查保证资金安全，避免重复支付和伪造回调。
P200: AI 智能赋能模块包含两条链路。文案润色链路：System Prompt + User Input -&gt; ChatClient.call() -&gt; 文本返回，润色结果自动填充至发布表单。导购链路：用户查询 -&gt; RagService 向量检索 Top 3 商品 -&gt; 上下文注入 System Prompt -&gt; DeepSeek 推理 -&gt; 返回文本和商品卡片。两条链路共享同一个 ChatClient Bean（openAiChatClient），通过 Spring AI 的 Qualifier 机制注入。模型选型上，对话能力使用 DeepSeek Chat，向量 Embedding 使用 Ollama 本地部署的 bge-m3 模型，两者通过 Spring AI 的抽象层解耦，后续可独立替换。
P201: 后台管控模块面向管理员。Dashboard 页面通过 ECharts 渲染近 7 日订单量折线图等统计图表。商品审核与公告管理属于常规增删改查操作。举报处理涉及跨表操作——处理举报时需同步更新物品状态并发送系统通知。

P395: 3.5 AI 大模型与向量知识库同步设计
P396: RAG 系统中存在一个核心工程问题：PgVector 中的向量数据与 MySQL 中的商品行记录如何保持一致性？商品上架后向量未同步将导致用户无法通过 AI 导购检索到该商品；商品售出后向量未删除则会导致用户检索到已售出商品，AI 推荐已售出商品会严重损害用户信任。这个问题在学术论文中通常不展开讨论，但在实际部署中每一个数据不一致都可能导致系统向用户输出错误推荐。
P397: SwapU 采用双轨同步策略解决该问题。第一轨为 RocketMQ 消息驱动的增量同步：MySQL 事务完成后构建 ItemSyncMessage 发送至 item-update-topic，ItemSyncListener 消费消息执行 ES 索引操作和 PgVector 向量同步。消息包含商品 ID 和操作类型（新增、更新、删除），消费端根据操作类型执行对应的索引和向量操作。MQ 不可用时降级为同步直连，确保数据最终一致。第二轨为全量同步：管理员通过 /admin/es/sync 接口触发 syncAllToEs()，遍历所有 status=1 的商品批量写入 ES 和 PgVector，用于系统初始化或灾难恢复。双轨策略的核心思想是“增量保实时，全量保正确”，增量同步保证日常运行时的数据时效性，全量同步作为兜底手段修复可能存在的不一致。RAG 向量双轨同步机制图如图 3-10 所示。

P401: 第4章 系统功能具体实现
P402: 本章节主要介绍实现系统重要功能的关键性方法和实现思路。本系统是基于Vue3+SpringBoot 的校园闲置物品循环利用系统，前端开发的语言主要采用的是 Vue 3，并使用 Element Plus UI 组件库快速搭建页面；后端采用 Spring Boot 3.2 框架，利用 MyBatis-Plus 简化数据库操作，通过 Spring AI 框架接入大语言模型。
P403: 4.1 系统功能实现的关键技术
P404: 4.1.1 前后端数据交互
P405: 在本系统中，前后端数据交互采用 Axios 网络请求库进行实现。具体实现思路是：首先使用 Axios 网络请求库创建了一个 HTTP 请求实例 request，并配置了两个参数。第一个参数是 baseURL，其属性设置为 /api，表示请求的基础 URL 为 /api，所有请求都会自动加上该前缀，由 Vite 开发服务器的 proxy 配置转发至后端 8080 端口。第二个参数是超时时间，设置为 60000 毫秒，以适应 AI 接口的较长响应时间，同时避免因网络波动导致请求无限等待。
P406: 此外，系统通过 Axios 拦截器统一处理鉴权与异常：请求拦截器从 localStorage 中获取 token 并直接注入 Authorization 请求头；响应拦截器捕获 401 状态码并自动清除本地存储跳转至登录页，捕获 403 状态码提示权限不足，其余错误统一以 Element Plus 的 Message 组件展示后端返回的错误信息。各业务模块无需重复编写鉴权与异常处理逻辑，显著降低了前端代码的冗余度。
P407: 4.1.2 JWT 无状态认证
P408: JWT 无状态认证方案的选型依据在于可扩展性。Session/Cookie 方案在水平扩展时需额外搭建共享 Session 存储（如 Redis Session），增加架构复杂度和单点故障风险。JWT 的去中心化特性使其天然适应分布式部署：token 自包含用户信息，服务端无需查询会话存储即可完成鉴权，任何实例均可独立验证 token 有效性。
P409: 签发逻辑集中在 JwtUtils 类中。密钥通过 Keys.hmacShaKeyFor() 从配置注入的 secret 派生，采用 HMAC-SHA256 算法，密钥长度满足安全要求。Claims 包含 userId、username、role 三个字段，过期时间 24 小时，与用户日常使用周期匹配。token 通过 Authorization 请求头传输，LoginInterceptor 在请求进入 Controller 前完成解析和校验，解析失败返回 401。
P410: JWT Claims 反序列化存在一个值得注意的问题：Integer 类型字段受 JVM 版本和 JSON 解析器影响，可能被还原为 Integer 或 Long。LoginInterceptor 解析 userId 时采用三层防御式类型判断（instanceof Integer -&gt; instanceof Long -&gt; toString 转换），确保类型安全。
P411: 4.1.3 后端访问操作数据库
P412: 在本系统中，后端访问数据库操作主要采用 MyBatis-Plus 提供的 BaseMapper 接口和 IService 接口，它们可用于简化数据库操作。系统采用 MVC 架构（控制层-服务层-数据访问层），可以实现关注点的分离、提高可重用性、实现松耦合。Controller 层通过 @RestController 和 @RequestMapping 注解定义 RESTful 接口，仅负责参数校验（@Validated）和响应封装（统一 Result 返回体）。Service 层通过 @Service 注解标记，承载全部业务逻辑，复杂查询使用 MyBatis-Plus 的 QueryWrapper 和 LambdaQueryWrapper 构建条件。数据访问层使用 MyBatis-Plus 提供的基础 Mapper 接口可以简化数据库操作的实现，减少开发工作量。
P413: 系统包含 15 个 Controller 和 13 个 ServiceImpl，基本一一对应。其中 ItemServiceImpl 依赖最密集，除 CRUD 外还承担 ES 同步和 RAG 向量写入；TradeOrderServiceImpl 封装订单创建、支付回调与状态流转逻辑；RagServiceImpl 封装向量检索逻辑，供 ChatController 调用。
P414: 4.2 系统功能实现详细介绍
P415: 4.2.1 商品检索功能实现
P416: 商品检索功能的核心实现在 ItemServiceImpl.search() 方法中。方法构建多层嵌套的 Elasticsearch DSL 查询：最外层 bool 必须 status=1（在售），可选 categoryId 过滤；关键词搜索通过 should 子句匹配 title 和 description 字段（IK 分词），支持通配符查询；高亮通过 HighlightQuery 配置 preTags 和 postTags 标签，前端接收到高亮标签后以醒目颜色渲染匹配关键词。搜索结果通过 SearchHits 转换为 MyBatis-Plus 的 Page 对象返回，保留了 ES 的相关性评分用于排序。
P417: Suggestion 自动补全功能采用 match_phrase_prefix 查询 title 字段实现，该方案无需额外的映射配置，与现有 text 类型字段天然兼容。搜索建议接口返回前 10 条匹配结果，前端在搜索框下方以下拉列表形式展示，用户点击后直接跳转至搜索结果页。
P418: 4.2.2 订单交易功能实现
P419: 订单与支付是整个系统中事务密度最高的模块。SwapU 的订单状态机设计如图 4-1 所示。
P421: 图 4-1 订单状态机流转图
P422: 状态机分布在两条执行路径中。正常路径：createOrder 通过 TransactionTemplate 创建 status=0 的订单并将商品状态更新为交易中；支付宝异步回调将 paymentStatus 更新为已支付、status 更新为进行中；买家确认收货后 status 更新为已完成、商品状态更新为已售出。异常路径：RocketMQ 延时消息触发 OrderTimeoutListener，检查订单仍为未支付状态则自动取消并恢复商品在售状态。两条路径覆盖了订单从创建到终态的所有可能走向，确保无论用户是否完成支付，系统状态都能正确收敛。
P423: createOrder() 的执行流程：参数校验 -&gt; Redisson 分布式锁对商品加锁 -&gt; 查询商品并校验状态 -&gt; TransactionTemplate.execute() 创建订单并更新商品状态 -&gt; 发送 RocketMQ 延时消息 -&gt; 释放锁。其中分布式锁是防止超卖的关键机制：锁的 key 为 item:lock:{itemId}，采用 tryLock(3, 10, SECONDS) 非阻塞式获取，等待 3 秒、持有 10 秒后自动释放。获取锁成功后方才执行商品状态校验和订单创建，确保同一商品在同一时刻只能被一个买家下单。若锁获取失败则返回&quot;当前下单人数过多，请稍后重试&quot;的提示。锁的释放放在 finally 块中，并通过 isHeldByCurrentThread() 判断避免释放其他线程持有的锁。TransactionTemplate 采用编程式事务而非声明式 @Transactional，是因为需要在业务校验通过后才开启事务，避免事务范围覆盖校验阶段导致长事务问题。订单创建成功后，商品状态由&quot;在售&quot;变更为&quot;交易中&quot;，同时向买家返回订单详情。
P424: 支付环节对接支付宝沙箱环境，后端构建 AlipayTradePagePayRequest 生成支付表单，前端通过表单提交跳转至支付宝收银台。支付完成后支付宝通过异步回调通知后端，后端在回调处理中执行 RSA 验签、幂等性检查和订单状态更新三个步骤，确保支付结果的真实性和一致性。RocketMQ 延时消息订单超时处理时序图如图 4-2 所示。

P427: 4.2.3 AI 智能导购功能实现
P428: AI 文案润色功能的 System Prompt 经过十余个版本迭代。初始版本存在三个问题：输出包含开场白（如“好的，我来帮你润色”），字数经常超标至 800 字以上，格式不统一（有时分段有时不分段）。最终版本的 Prompt 明确规定了三条约束：字数 500 以内、只返回文案本身不含客套话、分段清晰。temperature 参数设为 0.1 以降低随机性，确保相同输入的输出稳定性。经过多轮测试验证，最终版本的输出合格率从初始版本的不足 40% 提升至 95% 以上。文案润色功能的 System Prompt 原文如下：
P429: 你是一个专业的二手电商平台闲置物品文案写手。请根据用户提供的简单商品信息，帮他生成一段排版精美、生动有吸引力、分段清晰的商品详情描述文案（包含商品亮点、转手原因、新旧程度说明等，如果用户提供的信息较少，你可以根据商品名称进行合理的脑补丰富）。
P430: 要求：
P431: 1. 严格控制字数在 500 字以内，文案要精炼。
P432: 2. 注意：你输出的内容将直接填充到用户的输入框中，因此只需要返回文案本身，绝对不要包含任何客套话、开场白或结尾的提示语。
P433: 智能导购的实现复杂度显著高于文案润色。ChatController 的 /ai/chat 接口接收一个 Map 参数，从中提取 message（String）和 history（List）两个字段。调用链路分为三步。完整 RAG 工作流如图 4-3 所示。
P435: 图 4-3 RAG 智能导购工作流图
P436: 第一步，向量检索。RagService 调用 VectorStore.similaritySearch，配置 topK=3、similarityThreshold=0.0，使用 Ollama bge-m3 模型将用户查询转化为 1024 维向量，在 PgVector 中执行余弦相似度检索，返回最相关的 3 条商品文档。similarityThreshold 设为 0.0 是因为校园商品库规模有限，即使相似度不高也应返回结果交由模型判断，避免用户查询无果。topK=3 是在推荐多样性和上下文长度之间取的平衡点，3 条商品信息注入 Prompt 后仍留有足够的生成空间。
P437: 第二步，上下文拼接。systemPromptBuilder 首先追加角色定位与行为约束，然后判断检索结果是否为空——不为空时遍历列表将 title、price、description 注入相关商品上下文区块，并指示模型简短推荐而非长篇描述；为空时指示模型直接友好回复不强行推荐。历史对话截取最近 6 条（3 轮）加入上下文，使模型具备多轮对话能力。整个 Prompt 的拼接逻辑在 ChatController 中完成，RagService 仅负责向量检索，职责划分清晰。导购功能的 System Prompt 基础模板原文如下：
P438: 你是 SwapU 校园闲置交易平台的智能助手。你可以帮助用户寻找商品、解答交易问题或者进行日常闲聊。
P439: 如果用户的提问与寻找商品或买卖相关，我已经为你查到了以下几款平台现有的商品，你只需要顺着这些商品简短地做个推荐即可（不要自己编造不存在的商品）。
P440: 如果用户的提问与商品无关（比如日常问候、闲聊等），请直接友好地回复，不需要强行推荐商品。
P441: 当检索到相关商品时，Prompt 会动态追加商品上下文区块：
P442: 【相关商品上下文开始】
P443: 1. 标题: {title}, 价格: {price}, 描述: {description}
P444: 2. ...
P445: 【相关商品上下文结束】
P446: 请使用友好、热情的语气回答，简明扼要地引导用户查看下面展示的商品卡片。不要在文本里输出长篇大论的商品详情。
P447: 第三步，推理与封装。组装完成的 Prompt 通过 chatClient.call() 同步阻塞等待完整响应，返回结果包含 AI 文本和商品卡片数据，前端同时渲染文字回复和可点击的商品卡片。当前版本未采用流式输出，后续可切换为 chatClient.stream() 降低用户感知延迟。整个 RAG 调用链路的平均耗时约 3.5 秒，其中向量检索约 200 毫秒，Prompt 拼接约 50 毫秒，模型推理约 3 秒，瓶颈在模型推理环节。
P448: 导购场景的 Prompt 设计复杂度远高于润色。仅设定角色定位与格式约束并不充分，必须加入一条硬性指令：严禁编造不存在的商品。然而仅靠指令约束并不完全可靠——模型在缺乏具体商品信息时仍可能生成虚构推荐——因此还需将 PgVector 实时检索出的 Top 3 商品元数据注入 Prompt。这批检索数据构成了 RAG 管线的事实锚点，将模型输出约束在真实商品范围内，从机制层面抑制幻觉。

P587: 第6章 总结与展望
P588: 本文的核心工作围绕一个主题展开：如何将大语言模型在垂直电商场景中真正用好，而非仅仅接入。当前许多系统将大模型作为噱头简单接入，输出质量不可控、与业务数据脱节，用户体验反而不如传统方案。SwapU 的设计思路是让 AI 成为业务流程的有机组成部分，通过工程手段约束其输出行为，使其在降低用户操作门槛的同时保证推荐结果的可靠性。
P589: SwapU 是一套面向校园场景的闲置物品循环利用系统。基础交易链路依托 Spring Boot 与 MySQL 构建了完整闭环，涵盖商品发布、搜索检索、即时通讯、订单交易和支付结算等核心功能。核心工程价值集中在 AI 侧的整合与增强，而非简单的模型 API 调用。
P590: 通过 Spring AI 框架接入 DeepSeek Chat 和 Ollama bge-m3，实现了基于 RAG 的智能导购助手。工程价值较高的部分并非模型调用本身，而是围绕模型输出构建的两层约束机制，以及保证向量知识库与业务数据库一致性的双轨同步策略。第一层约束是 System Prompt 的迭代调优，文案润色功能经历十余版本修改才将输出行为收敛至稳定，解决了开场白冗余、字数超标和格式不统一三个核心问题。第二层约束是 RAG 的事实约束机制，导购助手每轮对话均先检索平台真实在售商品，将检索结果注入上下文，从源头限制模型的编造空间，这是区别于通用聊天机器人的关键设计。
P591: RocketMQ 延时消息在测试中验证了工程价值。超时自动取消机制在超时取消场景中表现正确，未出现状态不一致。编程式事务的选择确保了事务范围与业务逻辑的精确匹配，避免了长事务问题。Redisson 分布式锁在下单接口中有效防止了并发超卖，锁的获取与释放逻辑与事务边界配合清晰。双轨同步策略在商品库持续变动的场景下保证了向量数据与行记录的一致性，增量同步保证实时性，全量同步作为兜底修复手段。
P592: 尽管受到开发周期等因素的限制，系统的核心功能均已成功实现并达到设计预期，但仍存在以下改进空间：
P593: 第一，大模型推理延迟。AI 导购一次完整调用平均耗时 3.5 秒，当前为同步阻塞模式，用户需等待完整响应后才能看到结果。压缩延迟存在三个方向：同步改流式输出（chatClient.stream()），使模型逐字生成时前端即可渲染，降低用户感知延迟；向量检索与 Prompt 拼接并行化，当前两者是串行执行，可改为 CompletableFuture 并行调用；部署更轻量的推理模型，如 DeepSeek 的蒸馏版本或本地部署的小参数模型，牺牲少量质量换取推理速度。
P594: 第二，多模态检索能力缺失。当前搜索仅依赖文本信息，商品图片中的视觉信息未被利用。在校园二手交易场景中，许多用户更习惯“以图搜图”——看到同学使用的物品后拍照搜索同款。可行方案是通过 CLIP 等视觉-语言模型将商品图片转化为 Embedding 向量，与文本向量共同存入 PgVector，支持图片查询和图文混合查询。此外，还可引入商品图片的自动标签识别，如品牌、成色等，进一步丰富检索维度。
P595: 第三，移动端适配不足。当前系统仅针对 PC 端浏览器进行了界面优化，在移动端浏览时部分页面布局不够理想。考虑到校园用户大量使用手机访问，后续应采用响应式布局或开发独立的移动端页面，确保在手机和平板上的操作体验。此外，还可考虑开发微信小程序版本，利用微信的社交属性降低用户获取成本。
