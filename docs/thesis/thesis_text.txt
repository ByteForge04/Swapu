摘    要
高校闲置物品的周期性积压是普遍存在的现象，每学期末大量的教材、电子设备等物资由于学业阶段的转换而被淘汰，而新生群体每年都会产生刚性的采购需求，二者之间存在着结构性的错配。目前主流的二手交易平台（闲鱼、转转等）以全国大市场为架构基因，在校园场景下存在信息匹配效率低、交易摩擦大、发布门槛高三个方面的错位。同时大语言模型和检索增强生成（RAG）技术的发展给解决上述问题提供了新的技术途径。因此，开发一个面向校园场景的智能二手交易系统，对促进校园闲置物品的循环利用具有重要的现实意义。
本系统是基于 Spring Boot 3.2 和 Vue 3 架构设计开发的一个校园闲置物品循环利用系统，即 SwapU。数据层整合 MySQL、Redis、Elasticsearch 和 PgVector 四种存储引擎。主要完成以下功能模块：
（1）用户认证与个人中心。提供基于 BCrypt 加密的注册与登录功能，登录成功后签发 JWT Token 实现无状态认证。个人中心支持查看已发布商品、收藏列表和买入卖出订单。
（2）商品发布与检索。卖家可上传商品图片并填写信息完成发布，发布时支持 AI 一键文案润色以降低撰写门槛。检索方面基于 Elasticsearch 实现关键词全文检索与搜索建议，同时基于 RAG 管线提供 AI 智能导购，先从 PgVector 检索平台真实在售商品，再将检索结果注入上下文交由 DeepSeek 大模型生成推荐，从机制层面抑制幻觉输出。
（3）订单交易与支付。覆盖下单、支付宝沙箱支付、确认收货和取消订单的完整链路。下单接口通过 Redisson 分布式锁防止并发超卖，通过 RocketMQ 延时消息实现订单超时自动取消与库存回退。
（4）即时通讯与互动。买卖双方通过 WebSocket 建立实时私信通道，支持文本和图片消息的持久化与已读标记。商品详情页提供留言板功能，交易完成后支持双向评价。
（5）后台管理与数据看板。管理员可对商品、用户和举报进行审核管理，Dashboard 页面通过 ECharts 渲染平台运营数据图表。系统通过 AES-128 加密保护用户敏感字段，通过拦截器实现权限校验与越权防护。
关键词：校园二手交易；检索增强生成；大语言模型；Spring Boot；Vue 3
Abstract
The periodic accumulation of idle items in universities is a common phenomenon. At the end of each semester, a large number of textbooks, electronic devices, and other materials are discarded due to academic stage transitions, while new students consistently have rigid purchasing demands year after year, creating a structural mismatch between supply and demand. Existing mainstream second-hand trading platforms (such as Xianyu and Zhuanzhuan) are architecturally designed for a national market, resulting in three mismatches in campus scenarios: insufficient information matching efficiency, excessive transaction friction, and high publishing barriers. Meanwhile, the development of large language models and Retrieval-Augmented Generation (RAG) technologies has provided a new technical approach to addressing the above issues. Therefore, developing an intelligent second-hand trading system oriented toward campus scenarios is of significant practical value for promoting the recycling of idle items on campus.
This system is a campus idle item recycling system designed and developed based on Spring Boot 3.2 and Vue 3 architecture, named SwapU. The data layer integrates four storage engines: MySQL, Redis, Elasticsearch, and PgVector. The following functional modules are mainly accomplished:
(1) User authentication and personal center. It provides registration and login based on BCrypt encryption, and issues JWT Token upon successful login for stateless authentication. The personal center supports viewing published items, favorite lists, and buy/sell orders.
(2) Item publishing and retrieval. Sellers can upload item images and fill in information to complete publishing, with AI one-click copywriting polish available to lower the writing barrier. For retrieval, Elasticsearch-based full-text keyword search and search suggestions are provided, along with an AI intelligent shopping guide based on the RAG pipeline that first retrieves real listed products from PgVector and then injects the results into the context for the DeepSeek large model to generate recommendations, suppressing hallucinated output at the mechanism level.
(3) Order trading and payment. It covers the complete chain of order placement, Alipay sandbox payment, delivery confirmation, and order cancellation. The order placement interface prevents concurrent overselling through Redisson distributed locks, and implements automatic order cancellation with inventory rollback upon timeout through RocketMQ delayed messages.
(4) Instant messaging and interaction. Buyers and sellers establish real-time private messaging channels through WebSocket, supporting text and image message persistence and read status marking. The item detail page provides a message board, and two-way reviews are supported after transaction completion.
(5) Backend management and data dashboard. Administrators can review and manage items, users, and reports. The Dashboard page renders platform operational data charts through ECharts. The system protects user sensitive fields through AES-128 encryption and implements permission verification and unauthorized access prevention through interceptors.
Key Words: Campus Second-hand Trading; Retrieval-Augmented Generation; Large Language Model; Spring Boot; Vue 3

第1章 绪论
1.1 课题研究背景及意义
高校闲置物品的周期性积压，不是新出现的现象。每学期末，宿舍楼下成摞的考研书、没拆封的生活用品、用了一段时间的电子设备等，这些物资被淘汰的原因不是失去了使用价值，而是原主人的学业阶段发生了转变。一方面存在大量的闲置品不断产生，另一方面又是新生群体年复一年的刚性采购需求。二者的错配属于结构性的。同时，二手交易中买卖双方的信任缺失又加大了交易的难度[1]。因此，创建起一个高效快捷的校园二手交易生态，在资源循环利用以及绿色消费这两个方面都有着非常实际的意义。
闲鱼、转转为何不能很好地满足这个需求呢？它们的架构基因就是全国的大市场，也就是把算法放在前面来给用户推荐那些跨地区的好卖的商品，物流系统按照快递网络来安排。在校园里面出现三个方面的偏差，信息匹配的速度不够快，搜索的人在乎的是别人是不是住在同一条宿舍楼里，而不是全国信用分；交易摩擦大，校园里主要依靠当面交易，但是平台缺少同城面交的支持；发布门槛高，学生不愿意为短期闲置品写详细的描述。另外现有的平台全国大市场模式不能适应校园场景本地化的需求[2]，社区化闲置物品处理方式已经被证明可以提高资源流转速度[3]。校园交易有很强的季节性特点，每到学期结束的时候就是闲置物品大量出现的时候，每到学期开始的时候就是采购需要的时候，但是目前平台的推荐算法并没有考虑到这样的季节性。
针对上述信息匹配效率低、交易摩擦大、发布门槛高三个痛点，本课题的研究意义体现在两个层面。在应用层面，构建一个面向校园场景的专属交易平台，通过地理聚合缩短供需匹配路径、通过当面交易降低履约成本、通过智能化手段降低发布门槛，使校园闲置物品的流通从依赖社交群的碎片化撮合升级为结构化的高效匹配，切实促进资源的循环利用。在技术层面，大语言模型与检索增强生成技术的成熟为解决上述痛点提供了新的可能：AI 辅助写作可以降低卖家的文案撰写负担，语义检索可以弥补关键词匹配在理解口语化购物意图上的不足。然而，将大模型引入电商推荐场景面临幻觉问题的挑战——模型可能编造不存在的商品推荐，这在交易场景中是不可接受的。因此，如何利用 RAG 机制将模型输出约束在真实商品数据范围内，使 AI 推荐具备事实可追溯性，是本课题需要探索的关键技术问题。
1.2 国内外研究现状
二手交易已经形成比较成熟的范式。国外方面，eBay率先把分类信息数字化，创建起基于信誉评分的交易体系，其拍卖模式至今仍然是C2C交易的标杆；Craigslist以极简信息发布降低用户参与门槛，证明了轻量级信息撮合在本地化交易中的有效性；Poshmark则加入了社交互动、直播元素来提高平台黏性，开创了社交电商的先河。国内方面，闲鱼依靠支付宝的芝麻信用体系创建起C2C交易的信任机制，依靠社区化运营来提升用户的活跃度；转转在手机等3C品类上依靠质检、官方验机服务来实现差异化定位，解决了高价值二手商品信任缺失的痛点。上述平台共同的范式就是C2C或者C2B2C的大市场撮合模式，对校园场景的地理聚合、即时交付特性没有针对性的设计，在交互体验上也没有考虑到学生群体的使用习惯[4]。在学术研究方面，已经有关于基于Spring Boot的校园二手交易系统实现[14]，但是没有引入AI的能力，商品检索仍然依靠传统的关键词匹配，不能理解用户的语义化购物意图。
从技术演进的角度来说，二手电商平台的搜索能力经历了几个阶段的更新。最早使用的是数据库LIKE模糊查询，但是性能和精度都存在明显的问题；接着使用基于ElasticSearch的BM25算法配合协同过滤技术，大大提高了关键词匹配的精度和召回率，但是对于口语化、带有隐含意图的查询，语义理解层面还存在着明显的不足。传统搜索引擎只能找到包含“考研”关键词的商品标题，不能理解用户真正需要的是考研相关的教材和真题。大语言模型的出现给解决语义理解的困境赋予了新的技术途径[18]，但是直接使用大模型会出现幻觉问题，即模型会编造不存在的商品推荐，这在电商场景中是不可接受的。
RAG的核心思想就是先检索再生成，即在把用户查询交给大模型之前，先从外部知识库中检索出最相关的、真实的数据，再将这些数据作为参考上下文注入Prompt[16-17]，使模型的回答被约束在真实数据范围内。RAG 机制从根本上解决了大模型幻觉问题，使生成的内容有可追溯的事实依据。已有研究对垂直领域知识服务中的 RAG 问答系统进行构建，结合知识库和知识图谱，验证了 RAG 在专业领域中的有效性。针对电商智能客服场景，相关研究把 RAG 与多语言处理结合起来[6]，验证了检索增强在垂直电商中是可行的。这些研究给 SwapU 的 AI 导购功能设计提供理论支持和技术参考。
但是从学术验证到生产级部署，RAG系统会遇到很多工程上的难题。然而，商品库处于持续变动之中——新商品上架、旧商品下架、价格调整、交易成交等操作均会导致数据变更，如何保证向量数据库中的 Embedding 与关系型数据库中的行记录保持一致性，是 RAG 系统在实际部署中必须面对的工程难题。此外，消息传递过程中的丢失问题也需要考虑。这些问题在学术论文中通常不被展开讨论，但在实际部署中，每一处数据不一致都会导致系统向用户输出错误的推荐结果。本课题SwapU系统把注意力放在了这些工程落地的关键问题上。
本课题的 SwapU 系统是在上述背景之下提出的。与已有研究不同，SwapU 不满足于在实验室环境中验证 RAG 的可行性，而是将其部署到真实的校园交易场景中，面对商品库持续变动、用户查询口语化、模型输出不可控等实际工程挑战，逐一给出解决方案。SwapU 的设计目标是在保证基础交易功能完整的基础上，依靠 AI 的能力来降低发布门槛、提高导购效率，使校园二手交易由信息撮合模式转变为智能服务模式。
1.3 主要研究内容
本课题围绕"如何将 RAG 技术有效整合进校园二手交易场景"这一核心问题展开，研究内容分为三个层次：
（1）基础交易链路的构建。基于 Spring Boot 3.2 与 Vue 3 搭建前后端分离的校园闲置物品循环利用平台，覆盖商品发布、全文检索、即时通讯、订单交易与支付闭环等基础功能，确保从发布到交易的业务链路完整可用。
（2）AI 与业务的深度整合。发布侧研究如何通过 System Prompt 迭代调优使大模型输出收敛至稳定可控状态，降低卖家文案撰写门槛；导购侧研究如何基于 RAG 管线实现"先检索后生成"的推荐机制，从源头抑制模型幻觉，使推荐结果基于平台真实在售商品。
（3）工程落地的主要难题就是保证向量数据和关系数据的一致性（双轨同步办法），防止并发下单出现超卖情况，对敏感字段实行零侵入加密等工程上的困难，使系统在实际应用环境中保持正确的运行以及可靠的性能。

第2章 需求分析
2.1 功能需求分析
SwapU 校园闲置物品循环利用系统面向在校大学生，提供一个方便快捷的校园二手物品交易平台，使大学生可以方便地发布、检索、交易和沟通闲置物品。系统使用Spring Boot框架、Elasticsearch搜索引擎以及RAG检索增强生成技术来提供智能化的交易体验。功能上遵循发布、检索、交易、沟通的主线，保证每一个环节都有对应的功能支持。
本系统的用户角色通过 User 实体中的 role 字段区分，分为两类：普通用户（role=0，对应买卖双方）和系统管理员（role=1）。未登录用户通过 LoginInterceptor 的白名单机制仅可访问登录、注册和商品浏览页面。普通用户可执行商品发布、搜索、购买、即时通讯和 AI 交互等全部前台功能；管理员除拥有普通用户权限外，还可访问后台管理页面，执行商品审核、举报处理和公告发布等管理操作。
在上述功能需求的基础上，前台交易子系统的核心功能模块如下：
（1）用户认证与个人中心模块。提供基于 BCrypt 加密的注册与登录功能，登录成功后签发 JWT Token。个人中心支持查看已发布商品、收藏商品列表、买入/卖出订单的查询与管理。
（2）商品浏览与检索模块。首页用卡片式信息流展示在售商品，顶部有分类的导航。检索功能用 Elasticsearch 的 BoolQuery 复合查询，可以进行关键词搜索、分类筛选、高亮标注。
（3）AI 智能交互模块。有发布阶段的一键文案润色功能，还有独立的AI智能导购对话界面。导购助手要有两个主要的功能，就是理解用户自然语言的购物意图并给出有针对性的推荐，支持多轮对话上下文的记忆。
（4）即时通讯模块。买卖双方通过 WebSocket 建立实时私信通道，支持文本和图片消息，消息持久化至数据库。
（5）订单和支付流转模块。包含下单、支付拉起、状态查询、交易取消、确认收货等完整的流程。系统要保证同一商品不能被多个买家同时下单，下单接口要使用分布式锁对商品加锁，防止并发场景下的超卖问题。
后台管理子系统的主要功能就是数据监控以及平台治理：
（1）数据大盘模块。以 ECharts 图表展示平台关键运营指标——近 7 日订单量趋势、新增用户数、商品品类分布以及大模型接口调用频次。管理员可通过大盘快速了解平台运营状况，及时发现异常波动。
（2）内容审核模块。管理员可以对全平台商品、留言、用户举报等进行统一的管理，可以对违规商品执行强制下架操作。举报处理结果联动更新被举报对象的状态字段，同时向举报人发送系统通知，保证审核操作形成闭环。管理员还可以发布、管理系统公告，公告在用户端首页以弹窗或者横幅的形式展示。后台管理子系统用统一入口来对平台内容进行全流程的管控，保证交易生态的健康发展。
2.2 业务流程分析
SwapU 的业务核心可归纳为三条主线，三者之间存在清晰的顺序依赖：商品发布生成可检索的商品池，买家通过搜索或 AI 导购发现商品后产生购买意向进而生成订单，订单流转过程中买卖双方通过即时通讯贯穿始终。以下分别描述三条主线的业务流程。
卖家登录后进入发布页面，上传商品实物图片并选择分类。为降低文案撰写门槛，卖家可以在输入框里输入简要关键词，点击 AI 一键润色按钮，前端会把请求发送给 /ai/polish 接口，后端调用 DeepSeek Chat 生成结构化文案。文案生成后自动填充到描述输入框，卖家可以在此基础上做修改。确认后提交发布请求，后端先调用 AI 审核接口检测合规性，审核通过后商品进入在售状态，同时异步写入 Elasticsearch 索引和 PgVector 向量库，保证商品可以被搜索到以及被 AI 导购检索到。商品发布业务流程图如图 2.1 所示。

2.3 非功能需求分析
非功能性需求中，以下几个方面在本系统中具有刚性约束：
（1）响应时间需求。首页商品列表接口需在 200 毫秒内返回结果，搜索接口需在 500 毫秒内返回。下单接口因涉及数据库事务提交和消息发送，可放宽至 500 毫秒。AI 接口因包含大模型推理，响应时间在 3-10 秒之间，属于可接受范围。
（2）数据安全要求。手机号等敏感信息不能用明文保存起来，在数据库里做加密处理才是保证数据安全的一种办法[12]。如果数据库被泄露了，用户的隐私就会被直接暴露出来。本系统使用MyBatis-Plus的TypeHandler来加入AES-128加解密逻辑，在写入的时候自动加密，在读取的时候自动解密，业务代码不需要知道加密的过程。用户密码用BCrypt哈希保存起来，即使数据库被泄露了也找不到明文。所有的资源归属接口都要检查当前登录的用户ID，防止越权访问别人的数据。管理员接口通过 AdminInterceptor 校验用户角色，仅允许管理员身份访问后台管理功能，普通用户无权进入。
（3）系统可维护性需求。AI模块和业务模块之间需要通过接口约定来解耦，底层对话模型和向量库的具体选型对上层调用方来说应该保持透明，方便之后独立替换。系统要对Controller层进行全量审计日志。

2.4.1 技术可行性分析
SwapU 校园闲置物品循环利用系统技术上是可行的。本系统使用了比较成熟的技术栈，即Spring Boot3.2框架、Vue3前端框架、MySQL数据库管理，使用Axios进行前后端的数据交互，使用Spring AI框架接入大语言模型。这些技术已经被广泛应用，社区活跃度高，技术问题通常可以在StackOverflow等平台上找到解决方案[7-9]。Spring Boot自动配置、起步依赖大大降低了项目搭建成本，MyBatis-Plus条件构造器简化了动态SQL的编写，Spring AI ChatClient抽象层使得模型切换不需要修改业务代码。因此实现该系统的难度较小，依靠这些技术，系统可以达到预期的功能目标。
2.4.2 经济可行性分析
SwapU 校园闲置物品循环利用系统在经济可行性上占优。开发工具使用社区版IDE，框架、中间件都是开源免费的，部署只需要一台4核8GB的学生机就可以完成全部服务的运行，Ollama本地推理也不需要GPU支持。系统给学生群体提供一个方便的校园二手物品交易渠道，使用户以较低的成本得到想要的物品，减少资源的浪费，具有实际的经济价值和社会效益。

3.1 系统结构设计
SwapU 采用前后端分离的 B/S 架构。系统的运行时结构可拆分为五层，每一层的职责边界清晰、互不重叠。系统整体架构如图 3.1 所示。


图 3.1 系统分层架构图
表现层用 Vite 开发服务器代理 /api 前缀的动态请求，通过 Vite 的 proxy 配置转发到后端 8080 端口，前后端从部署层面完全解耦。前端用 Vue 3 的 Composition API 组织代码，Vue 的组件化架构和响应式数据绑定机制给复杂交互页面的开发赋予了高效的支撑[10]。状态管理使用 Pinia 替代 Vuex 来简化数据流，UI 组件库用 Element Plus。Axios 拦截器统一处理 token 注入和 401 跳转，各个业务模块不需要重复编写鉴权逻辑。
网关控制层在请求进入业务代码前完成鉴权与审计。LoginInterceptor 从 Authorization 请求头获取 token，调用 JwtUtils.validateToken() 解析 Claims 中的 userId、username、role，存入 Request 域供后续 Controller 使用。未登录用户访问受保护接口时返回 401 状态码，前端拦截器自动跳转至登录页。AdminInterceptor 在 LoginInterceptor 之后执行，校验 role=1 方可访问 /admin/** 路径，否则返回 403。GlobalLogAspect 通过 AOP 切面记录方法耗时与异常，为线上问题排查提供审计日志。
业务逻辑层包含 15 个 Controller 和 13 个 ServiceImpl，基本一一对应。Controller 层仅负责参数校验和响应封装，业务逻辑全部下沉至 Service 层。ItemServiceImpl 依赖最密集，除 CRUD 外还承担 ES 同步和 RAG 向量写入。TradeOrderServiceImpl 封装订单创建、支付回调与状态流转逻辑，是系统中事务密度最高的模块。RagServiceImpl 封装向量检索逻辑，供 ChatController 调用。ChatMessageService 处理 WebSocket 消息的持久化与已读状态更新。
数据持久层各存储引擎职责隔离。MySQL 承担 ACID 写入和关联查询，是系统的主数据源。Redis 在缓存和分布式锁等场景中具有成熟的应用实践[11]。本系统中 Redis 承载缓存热点商品详情以降低 MySQL 查询压力，同时通过 Redisson 分布式锁为下单接口提供防超卖保障。Elasticsearch 承担全文检索，通过 RocketMQ 异步同步商品数据，查询效率相比 MySQL LIKE 提升数个数量级。PgVector 承担向量语义检索，商品上架时异步写入 Embedding 向量，为 RAG 管线提供事实数据源。
基础设施层有四个外部系统。RocketMQ 维护 item-update-topic 和 order-timeout-topic 两个 Topic。支付宝沙箱提供PC网站支付，用异步回调来更新订单支付状态。Ollama 本地部署 bge-m3 模型提供 Embedding 服务。DeepSeek API 用 OpenAI 兼容格式提供 Chat 能力。

3.3 功能模块设计
系统从逻辑上分为四个模块，模块之间通过接口进行通信，不允许跨模块直接访问对方的数据库表。模块化设计使各个模块可以独立开发、测试，减少了系统的耦合度。下面分别对各个模块的功能进行阐述以及主要的设计决策。
基础支撑模块管理所有业务共同依赖的横切逻辑，其安全机制的设计参考了轻量级安全Web开发框架的研究思路[13]。密码用 BCrypt 哈希存储，登录验证时通过 PasswordEncoder.matches() 比对，避免明文传输和存储。JWT 签发逻辑封装在 JwtUtils 中，密钥通过 Keys.hmacShaKeyFor() 从配置注入的 secret 派生，采用 HMAC-SHA256 算法，Claims 包含 userId、username、role 三个字段，过期时间 24 小时。JWT 的无状态特性使系统天然支持水平扩展，无需共享 Session 存储。手机号用 AES-128 TypeHandler 加密存储，密钥配置在 application.yml 中，加解密过程对业务代码完全透明。
核心交易模块代码量最多，有商品、订单、支付这三个子模块。商品子模块有五种状态，分别是status=0待审核、status=1在售、status=2交易中、status=3已售出、status=4已下架。状态只能由后端在特定业务操作的时候来改变，前端不能直接更改状态字段。商品发布的时候会异步地写入到ES索引里以及PgVector向量里面去，保证可以被搜索到并且可以被导购。订单子模块也存在四种状态，即0待确认、1进行中、2已完成、3已取消。订单创建的时候用RocketMQ延时消息来达成30分钟超时自动取消的目的。支付子模块对接支付宝沙箱，异步回调的时候用RSA验签和幂等性检查来保证资金安全，防止重复支付或者伪造回调。
AI智能赋能模块有两条链路。文案润色链路为System Prompt + User Input -> ChatClient.call() -> 文本返回，润色结果自动填充到发布表单。导购链路为用户查询 -> RagService 向量检索 Top 3 商品 -> 上下文注入 System Prompt -> DeepSeek 推理 -> 返回文本和商品卡片。两条链路共享同一个ChatClientBean（openAiChatClient），利用Spring AI的Qualifier机制注入。模型选择上，对话能力使用DeepSeek Chat，向量Embedding使用Ollama本地部署的bge-m3模型，二者通过Spring AI的抽象层解耦，后续可独立替换。
后台管控模块面向管理员。Dashboard 页面用 ECharts 渲染近7日订单量折线图等统计图表。商品审核、公告管理属于常规的增删改查操作。举报处理需要跨表操作，即处理举报时要同时更新物品状态并发送系统通知。

3.5 AI 大模型与向量知识库同步设计
RAG 系统存在一个核心的工程问题：PgVector 中的向量数据与 MySQL 中的商品行记录如何保持一致性。商品上架之后向量没有同步就会造成用户不能通过AI导购检索到该商品；商品售出之后向量没有被删除，用户会检索到已售出的商品，AI推荐已售出的商品会严重损害用户的信任。这个问题在学术论文中一般不会被展开，但是实际部署中每一个数据不一致都会造成系统向用户输出错误推荐。
SwapU 用双轨同步来解决该问题。第一轨为RocketMQ消息驱动的增量同步，MySQL事务完成后生成ItemSyncMessage发送到item-update-topic，ItemSyncListener消费消息执行ES索引操作、PgVector向量同步。消息中包含商品ID和操作类型（新增、更新、删除），消费端根据操作类型执行对应的索引和向量操作。MQ不可用时降级为同步直连，保证数据最终一致。第二轨为全量同步，管理员通过 /admin/es/sync 接口触发 syncAllToEs()，遍历所有 status=1 的商品批量写入 ES 和 PgVector，用于系统初始化或者灾难恢复。双轨策略的核心思想是“增量保实时，全量保正确”，增量同步保证日常运行时的数据时效性，全量同步作为兜底手段修复可能存在的不一致。RAG 向量双轨同步机制图如图 3.10 所示。

第4章 系统功能具体实现
本章节主要介绍实现系统重要功能的关键性方法和实现思路。本系统是基于Vue3+SpringBoot 的校园闲置物品循环利用系统，前端开发的语言主要采用的是 Vue 3，并使用 Element Plus UI 组件库快速搭建页面；后端采用 Spring Boot 3.2 框架，利用 MyBatis-Plus 简化数据库操作，通过 Spring AI 框架接入大语言模型。
4.1 系统功能实现的关键技术
4.1.1 前后端数据交互
在本系统中，前后端数据交互采用 Axios 网络请求库进行实现。具体实现思路是：首先使用 Axios 网络请求库创建了一个 HTTP 请求实例 request，并配置了两个参数。第一个参数是 baseURL，其属性设置为 /api，表示请求的基础 URL 为 /api，所有请求都会自动加上该前缀，由 Vite 开发服务器的 proxy 配置转发至后端 8080 端口。第二个参数是超时时间，设置为 60000 毫秒，以适应 AI 接口的较长响应时间，同时避免因网络波动导致请求无限等待。
此外，系统用 Axios 拦截器统一处理鉴权和异常，请求拦截器从 localStorage 中取 token 并直接注入 Authorization 请求头，响应拦截器捕获 401 状态码并自动清除本地存储跳转至登录页，捕获 403 状态码提示权限不足，其余错误统一以 Element Plus 的 Message 组件展示后端返回的错误信息。各个业务模块不需要重复编写鉴权和异常处理的逻辑，大大减少了前端代码的冗余度。
4.1.2 JWT 无状态认证
JWT无状态认证方案的选择标准是可扩展性。Session/Cookie方案在水平扩展的时候要另外创建共享Session存储（Redis Session），造成架构复杂度以及单点故障的风险。JWT的去中心化特点使它天生适合分布式部署，token自带用户信息，服务端不需要查会话存储就可以完成鉴权，任何一个实例都可以单独验证token的有效性。
签发逻辑集中在 JwtUtils 类中。密钥通过 Keys.hmacShaKeyFor() 从配置注入的 secret 派生，采用 HMAC-SHA256 算法，密钥长度满足安全要求。Claims 包含 userId、username、role 三个字段，过期时间 24 小时，与用户日常使用周期匹配。token 通过 Authorization 请求头传输，LoginInterceptor 在请求进入 Controller 前完成解析和校验，解析失败返回 401。
JWT Claims反序列化存在一个值得注意的问题，Integer类型字段受JVM版本以及JSON解析器的影响，可以被还原成Integer或者Long。LoginInterceptor解析userId的时候用的是三层防御式类型判断（instanceof Integer -> instanceof Long -> toString转换），保证类型安全。
4.1.3 后端访问操作数据库
本系统后端数据库访问操作主要使用MyBatis-Plus提供的BaseMapper接口和IService接口来简化数据库操作。系统采用MVC架构（控制层-服务层-数据访问层），可以实现关注点的分离、提高可重用性、实现松耦合。Controller层用@RestController和@RequestMapping注解定义RESTful接口，只做参数校验（@Validated）和响应封装（统一Result返回体）。Service层用@Service注解标记，承担所有的业务逻辑，复杂的查询用MyBatis-Plus的QueryWrapper和LambdaQueryWrapper来构建条件。数据访问层使用MyBatis-Plus提供的BaseMapper接口可以简化数据库操作的实现，减少开发工作量。
系统包含 15 个 Controller 和 13 个 ServiceImpl，基本一一对应。其中 ItemServiceImpl 依赖最密集，除 CRUD 外还承担 ES 同步和 RAG 向量写入；TradeOrderServiceImpl 封装订单创建、支付回调与状态流转逻辑；RagServiceImpl 封装向量检索逻辑，供 ChatController 调用。
4.2 系统功能实现详细介绍
4.2.1 商品检索功能实现
商品检索功能主要实现于ItemServiceImpl.search()方法。方法创建多层嵌套的Elasticsearch DSL查询，最外层bool必须status=1（在售），可以设置categoryId过滤，关键词搜索使用should子句匹配title和description字段（IK分词），支持通配符查询，高亮使用HighlightQuery配置preTags和postTags标签，前端收到高亮标签后用醒目的颜色渲染匹配的关键词。搜索结果用SearchHits转换成MyBatis-Plus的Page对象返回，保持了ES相关的评分用于排序。
建议自动补全功能用match_phrase_prefix来查title字段，不需要其他的映射设置，和现有的text类型字段自然地相容。搜索建议接口返回前10个匹配的结果，在搜索框下面以下拉列表的形式出现，用户点击之后就会直接跳到搜索结果页面上。
4.2.2 订单交易功能实现
订单与支付是整个系统中事务密度最高的模块。SwapU 的订单状态机设计如图 4.1 所示。

图 4.1 订单状态机流转图
状态机分布在两条执行路径中。正常路径createOrder用TransactionTemplate创建status=0的订单，将商品状态设为交易中，支付宝异步回调将paymentStatus设为已支付，status设为进行中，买家确认收货后status设为已完成，商品状态设为已售出。异常路径RocketMQ延时消息触发OrderTimeoutListener，订单未支付状态则自动取消并恢复商品在售状态。两条路径覆盖了订单从创建到终态的所有可能走向，保证用户不管是否完成支付，系统状态都能正确收敛。
createOrder() 的执行流程：参数校验 -> Redisson 分布式锁对商品加锁 -> 查询商品并校验状态 -> TransactionTemplate.execute() 创建订单并更新商品状态 -> 发送 RocketMQ 延时消息 -> 释放锁。其中分布式锁是防止超卖的关键机制：锁的 key 为 item:lock:{itemId}，采用 tryLock(3, 10, SECONDS) 非阻塞式获取，等待 3 秒、持有 10 秒后自动释放。获取锁成功后方才执行商品状态校验和订单创建，确保同一商品在同一时刻只能被一个买家下单。若锁获取失败则返回"当前下单人数过多，请稍后重试"的提示。锁的释放放在 finally 块中，并通过 isHeldByCurrentThread() 判断避免释放其他线程持有的锁。TransactionTemplate 采用编程式事务而非声明式 @Transactional，是因为需要在业务校验通过后才开启事务，避免事务范围覆盖校验阶段导致长事务问题。订单创建成功后，商品状态由"在售"变更为"交易中"，同时向买家返回订单详情。
支付环节对接支付宝沙箱环境，后端创建AlipayTradePagePayRequest生成支付表单，前端用表单提交跳转到支付宝收银台。支付完成后支付宝通过异步回调通知后端，在回调处理中完成RSA验签、幂等性检查、订单状态更新这三个步骤，保证支付结果的真实性和一致性。RocketMQ 延时消息订单超时处理时序图如图 4.2 所示。

4.2.3 AI 文案润色与智能导购功能实现
AI 文案润色功能的 System Prompt 经过十余个版本迭代。初始版本存在三个问题，输出包含开场白（如“好的，我来帮你润色”），字数经常超标至 800 字以上，格式不统一（有时分段有时不分段）。最终版本的 Prompt 明确规定了三条约束，字数 500 以内、只返回文案本身不含客套话、分段清晰。temperature 参数设为 0.1 以降低随机性，保证相同输入的输出稳定。经过多次测试之后，最终版本的输出合格率由最初的不足 40% 提高到了 95% 以上。文案润色功能的 System Prompt 原文如下：
你是一个专业的二手电商平台闲置物品文案写手。请根据用户提供的简单商品信息，帮他生成一段排版精美、生动有吸引力、分段清晰的商品详情描述文案（包含商品亮点、转手原因、新旧程度说明等，如果用户提供的信息较少，你可以根据商品名称进行合理的脑补丰富）。
要求：
1. 严格控制字数在 500 字以内，文案要精炼。
2. 注意：你输出的内容将直接填充到用户的输入框中，因此只需要返回文案本身，绝对不要包含任何客套话、开场白或结尾的提示语。
智能导购的实现复杂度显著高于文案润色。ChatController 的 /ai/chat 接口接收一个 Map 参数，从中提取 message（String）和 history（List）两个字段。调用链路分为三步。完整 RAG 工作流如图 4.3 所示。

图 4.3 RAG 智能导购工作流图
第一步，向量检索。RagService 调用 VectorStore.similaritySearch，配置 topK=3、similarityThreshold=0.0，使用 Ollama bge-m3 模型将用户查询转化为 1024 维向量，在 PgVector 中执行余弦相似度检索，返回最相关的 3 条商品文档。similarityThreshold 设为 0.0 是因为校园商品库规模有限，即使相似度不高也应返回结果交由模型判断，避免用户查询无果。topK=3 是在推荐多样性和上下文长度之间取的平衡点，3 条商品信息注入 Prompt 后仍留有足够的生成空间。
第二步，上下文拼接。systemPromptBuilder 首先追加角色定位与行为约束，然后判断检索结果是否为空——不为空时遍历列表将 title、price、description 注入相关商品上下文区块，并指示模型简短推荐而非长篇描述；为空时指示模型直接友好回复不强行推荐。历史对话截取最近 6 条（3 轮）加入上下文，使模型具备多轮对话能力。整个 Prompt 的拼接逻辑在 ChatController 中完成，RagService 仅负责向量检索，职责划分清晰。导购功能的 System Prompt 基础模板原文如下：
你是 SwapU 校园闲置交易平台的智能助手。你可以帮助用户寻找商品、解答交易问题或者进行日常闲聊。
如果用户的提问与寻找商品或买卖相关，我已经为你查到了以下几款平台现有的商品，你只需要顺着这些商品简短地做个推荐即可（不要自己编造不存在的商品）。
如果用户的提问与商品无关（比如日常问候、闲聊等），请直接友好地回复，不需要强行推荐商品。
当检索到相关商品时，Prompt 会动态追加商品上下文区块：
【相关商品上下文开始】
1. 标题: {title}, 价格: {price}, 描述: {description}
2. ...
【相关商品上下文结束】
请使用友好、热情的语气回答，简明扼要地引导用户查看下面展示的商品卡片。不要在文本里输出长篇大论的商品详情。
第三步，推理和封装。组装完成的Prompt用chatClient.call()同步阻塞等待完整响应，返回结果包含AI文本和商品卡片数据，前端同时渲染文字回复和可点击的商品卡片。目前版本不支持流式输出，之后可以使用chatClient.stream()来降低用户的感知延迟。RAG调用链路平均耗时为3.5秒，向量检索占200毫秒，Prompt拼接占50毫秒，模型推理占3秒，瓶颈为模型推理环节。
导购场景的Prompt设计复杂度远大于润色。仅设定角色定位和格式约束是不够的，必须加上一条硬性指令：不能编造不存在的商品。但是仅靠指令约束并不完全可靠，模型在缺少具体商品信息的情况下仍然可以生成虚构的推荐，因此还需要将 PgVector 实时检索出的 Top 3 商品元数据注入到 Prompt 中。这批检索数据成为 RAG 管线的现实锚点，把模型输出的限制限定在真实的商品之内，从机制上抑制幻觉。

第6章 总结与展望
本文的核心工作围绕一个主题展开：如何将大语言模型在垂直电商场景中真正用好，而非仅仅接入。当前许多系统将大模型作为噱头简单接入，输出质量不可控、与业务数据脱节，用户体验反而不如传统方案。SwapU 的设计思路是让 AI 成为业务流程的有机组成部分，通过工程手段约束其输出行为，使其在降低用户操作门槛的同时保证推荐结果的可靠性。
SwapU 是一套面向校园场景的闲置物品循环利用系统。基础交易链路依托 Spring Boot 与 MySQL 构建了完整闭环，涵盖商品发布、搜索检索、即时通讯、订单交易和支付结算等核心功能。核心工程价值集中在 AI 侧的整合与增强，而非简单的模型 API 调用。
利用Spring AI框架连接DeepSeek Chat和Ollama bge-m3，创建出基于RAG的智能导购助手。工程价值高的部分不是模型调用本身，而是围绕模型输出建立的两层约束机制，以及保证向量知识库与业务数据库一致性的双轨同步策略。第一层约束是System Prompt的迭代调优，文案润色功能经过十余个版本的修改，才使输出行为收敛到稳定状态，解决了开场白冗余、字数超标、格式不统一这三个核心问题。第二层约束为RAG事实约束机制，导购助手每轮对话先检索出平台真实的在售商品，把检索到的结果作为上下文内容，从源头上限制模型的编造空间，这是不同于通用聊天机器人的设计特点。
RocketMQ延时消息经由测试来证明它的工程意义。超时自动取消机制对于超时取消情况来说是正确的，没有发生状态不一致的情况。编程式事务的选择可以保证事务范围和业务逻辑的准确对应，防止出现长事务的问题。Redisson分布式锁对于下单接口来说起到了很好的防止并发超卖的作用，锁的获得以及释放的过程同事务边界的配合十分清楚。双轨同步策略对于商品库不断变化的情况而言可以保证向量数据和行记录的一致性，增量同步可以保证实时性，全量同步就是兜底的修复办法。
虽然受到开发周期等因素的影响，系统的核心功能已经实现并达到了设计的预期，但是还存在着以下的改进空间：
第一，大模型推理延迟。AI 导购一次完整调用平均耗时 3.5 秒，当前为同步阻塞模式，用户需等待完整响应后才能看到结果。压缩延迟存在三个方向：同步改流式输出（chatClient.stream()），使模型逐字生成时前端即可渲染，降低用户感知延迟；向量检索与 Prompt 拼接并行化，当前两者是串行执行，可改为 CompletableFuture 并行调用；部署更轻量的推理模型，如 DeepSeek 的蒸馏版本或本地部署的小参数模型，牺牲少量质量换取推理速度。
第二，缺乏多模态检索能力。目前的搜索只用文本信息，商品图片中所包含的视觉信息没有被利用起来。校园二手交易场景下，很多用户会用“以图搜图”，即看到同学使用的物品之后就拍照搜索同款。可行的方法就是用CLIP等视觉-语言模型把商品图片转化为Embedding向量，把文本向量一起存入PgVector，可以完成图片查询以及图文混合查询。另外还可以使用商品图片自动标签识别，比如品牌、成色等来丰富检索维度。
第三点就是移动端适配的问题。现在系统只给PC端浏览器做了界面的改善，但是对于移动端浏览来说，有些页面的布局并不是很好。由于校园里很多学生都是用手机来访问的，所以以后应该用响应式布局或者创建专门的移动端页面来保证手机和平板上操作的良好体验。另外还可以创建一个微信小程序版本，借助微信的社会属性来减少用户的获取成本。
