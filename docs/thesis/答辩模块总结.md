# SwapU 答辩模块总结

## 一、登录模块（基础支撑模块）

### 1.1 模块架构

| 层次 | 类名 | 职责 |
|------|------|------|
| Controller | UserController | 登录/注册/用户信息接口 |
| Service | UserServiceImpl | 登录校验 + BCrypt + JWT 签发 |
| Utils | JwtUtils | JWT 生成、解析、类型防御 |
| Interceptor | LoginInterceptor | 请求级鉴权，解析 token 存入 Request |
| Interceptor | AdminInterceptor | 管理员权限校验（role==1） |
| 前端 | request.js | Axios 实例 + 拦截器注入 token |

### 1.2 登录流程

```
前端提交 username + password
    ↓
UserServiceImpl.login()
    ├── QueryWrapper 查询用户
    ├── BCrypt PasswordEncoder.matches() 比对密码
    │   └── 兼容明文：若 BCrypt 匹配失败，尝试明文比对 + 自动升级为 BCrypt 哈希
    ├── 校验用户状态（status==1）
    └── JwtUtils.generateToken(userId, username, role)
    ↓
返回 token → 前端存入 localStorage
```

### 1.3 JWT 认证方案

- **算法**：HMAC-SHA256，密钥通过 `Keys.hmacShaKeyFor()` 从配置派生
- **Claims**：userId、username、role
- **有效期**：24 小时
- **无 Bearer 前缀**：token 直接存入 localStorage，Axios 请求拦截器直接注入 `Authorization` 头

### 1.4 JWT Claims 反序列化三层防御

```java
// userId 可能被还原为 Integer 或 Long（受 JVM 版本和 JSON 解析器影响）
Object userIdObj = claims.get("userId");
if (userIdObj instanceof Integer) {
    userId = ((Integer) userIdObj).longValue();
} else if (userIdObj instanceof Long) {
    userId = (Long) userIdObj;
} else {
    userId = Long.parseLong(userIdObj.toString());
}
```

### 1.5 Axios 拦截器

- **请求拦截器**：从 localStorage 读取 token → 注入 `Authorization` 请求头（无 Bearer 前缀）
- **响应拦截器**：401 → 清除存储 + 跳转登录页；403 → 提示权限不足；其他错误 → 显示错误信息

### 1.6 BCrypt 密码哈希

- 注册时：`passwordEncoder.encode(plaintext)` → 存储哈希值
- 登录时：`passwordEncoder.matches(rawPassword, encodedPassword)`
- **自动升级**：若 BCrypt 匹配失败但明文相等，自动将明文升级为 BCrypt 哈希存储

### 1.7 AES TypeHandler 零侵入加密

- User 实体的 phone 字段标注 `@TableField(typeHandler = AesEncryptTypeHandler.class)`
- 写入时自动 AES-128 加密（Base64 密文存入数据库）
- 读取时自动解密（API 返回明文）
- **零侵入**：业务代码无需关心加解密，MyBatis-Plus 在 JDBC 层自动处理

### 1.8 答辩追问

| 追问 | 应答要点 |
|------|---------|
| 为什么选 JWT 而不是 Session？ | JWT 无状态，天然支持分布式部署，不需要共享 Session 存储，减少架构复杂度和单点故障风险 |
| 为什么不用 Bearer 前缀？ | 项目规模较小，简化前端处理逻辑；Bearer 是规范建议但非强制要求 |
| token 过期怎么办？ | 当前方案是前端检测 401 后跳转登录页重新登录；后续可引入 Refresh Token 机制 |
| AES 密钥怎么管理？ | 密钥配置在 application.yml 中，生产环境应通过环境变量注入，不硬编码在代码中 |

## 二、交易模块（核心交易模块）

### 2.1 模块架构

| 层次 | 类名 | 职责 |
|------|------|------|
| Controller | TradeOrderController | 订单 CRUD 接口 + 限流 |
| Controller | PaymentController | 支付宝沙箱支付 + 回调 |
| Service | TradeOrderServiceImpl | 订单全生命周期业务逻辑 |
| Listener | OrderTimeoutListener | MQ 延迟消息消费，超时自动取消 |

### 2.2 订单状态机

```
0(待确认/待支付) → 1(进行中) → 2(已完成)
  ↓              ↓
3(已取消)     3(已取消)
```

物品状态联动：

```
0(待审核) → 1(在售) → 2(交易中) → 3(已售出)
                       ↑    ↓
                      1(在售)  ← 取消时回退
```

- **下单**：物品 1(在售) → 2(交易中)
- **取消**：物品 2(交易中) → 1(在售)
- **完成**：物品 2(交易中) → 3(已售出)

### 2.3 下单流程（createOrder）

```
买家下单请求
    ↓
Controller: Redisson 限流（每分钟3次/用户）
    ↓
Service: Redisson 分布式锁（key=item:lock:{itemId}）
    ↓ tryLock(3, 10, SECONDS)
TransactionTemplate 编程式事务
    ├── 1. 校验物品（存在？在售？非自己？）
    ├── 2. 校验金额（前端传入 vs 物品价格）
    ├── 3. 填充订单（UUID订单号、sellerId、status=0）
    ├── 4. 保存订单 + 物品状态→2(交易中)
    ├── 5. 通知卖家（系统通知）
    └── 6. 发送 RocketMQ 延迟消息（30分钟超时）
    ↓
finally: isHeldByCurrentThread() → unlock()
```

**为什么用 TransactionTemplate 而不是 @Transactional？**

> `@Transactional` 在方法入口就开启事务，而获取分布式锁的 `tryLock(3,10)` 最多等待 3 秒。如果先开事务再等锁，会造成长事务问题——数据库连接被占用但无事可做。TransactionTemplate 在获取锁之后才开启事务，事务范围最小化。

**分布式锁 key 设计：`item:lock:{itemId}`**

> 粒度是物品级别，不是全局锁。不同物品可以并发下单，只有同一物品的并发请求才互斥。

### 2.4 支付流程（支付宝沙箱）

```
买家点击支付 → GET /pay/alipay?orderId=xxx
    ↓
PaymentController: 构造 AlipayTradePagePayRequest
    ├── out_trade_no = orderNo（UUID）
    ├── total_amount = 订单金额
    ├── notifyUrl = 异步回调地址
    └── returnUrl = 同步回调地址
    ↓
返回支付宝支付页面 HTML（pageExecute）
    ↓
用户完成支付
    ↓
┌─────────────────────────────────────────┐
│  异步回调 POST /pay/notify（生产环境）    │
│    → RSA 验签 → TRADE_SUCCESS?          │
│    → paySuccess(orderNo, tradeNo, time)  │
├─────────────────────────────────────────┤
│  同步回调 GET /pay/return（开发环境兜底）  │
│    → RSA 验签 → 同样调用 paySuccess      │
│    → 重定向到订单详情页                    │
└─────────────────────────────────────────┘
```

**paySuccess 幂等性设计**：

```java
if (order.getPaymentStatus() != null && order.getPaymentStatus() == 1) {
    return; // 已支付，直接返回
}
```

> 同步回调和异步回调都可能触发 paySuccess，幂等检查确保多次调用安全。状态更新：paymentStatus=1，status=0→1(进行中)。

### 2.5 超时自动取消（RocketMQ 延迟消息）

```
createOrder 时发送延迟消息
    → topic: order-timeout-topic
    → DelayTimeLevel: 16（= 30分钟）
    ↓
30分钟后 OrderTimeoutListener 消费
    → 检查订单 status==0 且 paymentStatus!=1
    → 调用 cancelOrder 自动取消
    → 物品状态恢复为 1(在售)
```

**为什么用延迟消息而不是定时任务？**

> 定时任务需要轮询数据库，间隔内可能有延迟；延迟消息是事件驱动，精确到秒级触发，且不占用数据库查询资源。

### 2.6 N+1 查询优化（listMyOrders）

```java
// 收集所有 itemId 和 userId
Set<Long> itemIds = list.stream().map(TradeOrder::getItemId).collect(toSet());
Set<Long> userIds = ...; // buyerId + sellerId

// 批量查询，一次 SQL 搞定
Map<Long, Item> itemMap = itemMapper.selectBatchIds(itemIds)...;
Map<Long, User> userMap = userMapper.selectBatchIds(userIds)...;

// 内存中填充
for (TradeOrder order : list) {
    order.setItem(itemMap.get(order.getItemId()));
    order.setBuyer(userMap.get(order.getBuyerId()));
    order.setSeller(userMap.get(order.getSellerId()));
}
```

> 将 N+1 次查询优化为 3 次（1次订单 + 1次物品 + 1次用户）。

### 2.7 答辩追问

| 追问 | 应答要点 |
|------|---------|
| 为什么分布式锁不用 Redis SETNX？ | Redisson 封装了看门狗续期机制，避免锁过期但业务未完成；且提供 isHeldByCurrentThread() 防止误删他人锁 |
| 锁的粒度为什么是物品级？ | 全局锁会串行化所有下单请求，吞吐量极低；物品级锁只对同一物品互斥，不同物品可并发 |
| tryLock 等待3秒会不会太长？ | 3秒是最大等待时间，实际获取到锁会立即返回；拿不到说明竞争激烈，返回"请稍后重试"更友好 |
| MQ 发送失败怎么办？ | 代码中 catch 异常仅 warn 日志，订单仍创建成功。可补充定时任务兜底扫描 |
| 为什么支付用沙箱？ | 毕设项目无法申请正式支付接口，支付宝沙箱提供完整支付流程模拟 |
| 同步回调为什么也调 paySuccess？ | 开发环境没有公网 IP，异步通知无法到达。paySuccess 有幂等校验，双重调用安全 |
| 线下交易和线上交易的区别？ | 线上：下单→买家支付→自动变进行中→买家确认收货→完成；线下：下单→卖家确认→进行中→买家确认→完成（跳过支付） |

## 三、聊天模块（核心交易模块）

### 3.1 模块架构

| 层次 | 类名 | 职责 |
|------|------|------|
| WebSocket | ChatWebSocketEndpoint | 实时消息收发（@ServerEndpoint） |
| Config | WebSocketConfig | 注册 ServerEndpointExporter Bean |
| Controller | ChatMessageController | HTTP 接口：历史记录、联系人、已读标记、未读数 |
| Service | ChatMessageServiceImpl | 聊天业务逻辑 |
| Entity | ChatMessage | 消息实体（chat_message 表） |

### 3.2 消息实体

```
chat_message 表
├── msgId       (自增主键)
├── senderId    (发送者 userId)
├── receiverId  (接收者 userId)
├── content     (消息内容：文本/图片URL)
├── msgType     (1=文本, 2=图片, 3=商品卡片)
├── relatedId   (关联ID，如商品卡片对应的 itemId)
├── isRead      (0=未读, 1=已读)
└── createdAt   (创建时间)
```

### 3.3 WebSocket 连接与鉴权

```
前端: new WebSocket(`ws://{host}/ws/chat/{userId}?token={jwt}`)
    ↓
@OnOpen: 从 URL 参数提取 token
    ↓
JwtUtils.getUserId(token) 解析出 tokenUserId
    ↓
校验 tokenUserId == URL 中的 userId（防止伪造身份）
    ↓
通过 → webSocketMap.put(userId, this) → 连接建立
失败 → session.close(VIOLATED_POLICY)
```

**鉴权设计要点**：
- WebSocket 无法通过 Header 传 token，通过 URL query 参数 `?token=xxx` 传递
- 校验 token 中的 userId 与 URL 路径中的 userId 一致，防止身份伪造

### 3.4 连接管理

```java
private static ConcurrentHashMap<Long, ChatWebSocketEndpoint> webSocketMap = new ConcurrentHashMap<>();
```

- Key：userId，Value：该用户的 WebSocket Session 实例
- 同一用户重复连接会覆盖旧连接
- 在线计数 onlineCount 用 synchronized 方法保证原子性

### 3.5 消息收发流程

```
发送方前端: ws.send(JSON.stringify({receiverId, content, msgType}))
    ↓
@OnMessage: 解析 JSON → ChatMessage
    ├── 设置 senderId, createdAt, isRead=0
    ├── chatMessageMapper.insert() → 持久化到数据库
    └── 查找接收方是否在线
        ├── 在线 → webSocketMap.get(receiverId).sendMessage() → 实时推送
        └── 不在线 → 仅落库，等对方上线后通过 HTTP 拉取历史记录
```

**关键设计：先落库再转发**

> 消息先写入 chat_message 表，再尝试实时推送。即使接收方不在线，消息也不会丢失——下次打开聊天时通过 HTTP 接口拉取历史记录。

### 3.6 心跳保活机制

```
前端: 每 20 秒发送 "ping"
    ↓
@OnMessage: 判断 message.equals("ping") → 回复 "pong"
```

- 防止代理服务器/负载均衡器因空闲超时断开连接
- 前端检测到 onclose 事件后，3 秒自动重连

### 3.7 HTTP 历史记录接口

| 接口 | 方法 | 功能 |
|------|------|------|
| `/chat/history/{contactId}` | GET | 获取与某联系人的历史消息（按时间正序） |
| `/chat/read/{contactId}` | POST | 标记某联系人的消息为已读 |
| `/chat/contacts` | GET | 获取联系人列表（含未读数、最后消息时间） |
| `/chat/unread-count` | GET | 获取全局未读消息总数 |

### 3.8 联系人列表（自定义 SQL）

```sql
SELECT sender_id, MAX(created_at) as last_time,
       COUNT(CASE WHEN is_read = 0 THEN 1 END) as unread_count
FROM chat_message WHERE receiver_id = #{userId} GROUP BY sender_id
UNION
SELECT receiver_id as sender_id, MAX(created_at) as last_time, 0 as unread_count
FROM chat_message WHERE sender_id = #{userId} GROUP BY receiver_id
ORDER BY last_time DESC
```

> 第一段查"别人发给我的"（有未读数），第二段查"我发给别人的"（未读数为0）。UNION 合并后去重。

### 3.9 WebSocket Bean 注入问题

```java
private static ChatMessageMapper chatMessageMapper;

@Autowired
public void setChatMessageMapper(ChatMessageMapper chatMessageMapper) {
    ChatWebSocketEndpoint.chatMessageMapper = chatMessageMapper;
}
```

> `@ServerEndpoint` 每次连接创建新实例（Jakarta WebSocket 规范），新实例不由 Spring 管理。通过静态变量 + @Autowired setter，在 Spring 初始化时将 Bean 赋值给 static 字段。

### 3.10 答辩追问

| 追问 | 应答要点 |
|------|---------|
| 为什么用 WebSocket 而不是轮询？ | 轮询延迟高、浪费带宽；WebSocket 双向实时通信，消息延迟毫秒级 |
| 为什么不用 STOMP/SockJS？ | STOMP 适合多人聊天室；SwapU 是一对一私信，原生 WebSocket 足够简单高效 |
| 消息丢失怎么办？ | 先落库再转发，消息持久化后才推送。不在线时下次通过 HTTP 拉取 |
| 图片消息怎么实现的？ | el-upload 上传图片到 /api/common/upload 返回 URL，通过 WebSocket 发送 msgType=2 的消息 |
| 多设备登录怎么办？ | 当前是一对一映射会覆盖旧连接；如需多设备支持可改为 userId→Set<Session> |

## 四、商品模块（核心交易模块）

### 4.1 模块架构

| 层次 | 类名 | 职责 |
|------|------|------|
| Controller | ItemController | 商品 CRUD + 搜索 + 建议接口 |
| Service | ItemServiceImpl | 商品业务逻辑 + ES 同步 + RAG 向量写入 |
| Entity | Item | MySQL 商品实体 |
| ES Doc | ItemDoc | Elasticsearch 文档实体 |
| Listener | ItemSyncListener | MQ 消费者，执行 ES 增量同步 |

**ItemServiceImpl 是系统中依赖最密集的 Service**——除 CRUD 外还承担 ES 同步和 RAG 向量写入。

### 4.2 商品五态状态机

```
0(待审核) ──AI审核通过──→ 1(在售) ──买家下单──→ 2(交易中) ──确认收货──→ 3(已售出)
  │                        │  ↑                    │
  │                     下架│  │取消订单              │
  │                        ↓  │                    │
  └──AI审核不通过──→ 4(已下架) └────────────────────┘
```

- **发布时**：AI 审核通过 → status=1；AI 审核不通过 → status=4；AI 异常 → status=0(待人工审核)
- **修改后**：status 重置为 0，需重新审核
- **下架/删除**：从 ES 中删除索引

### 4.3 AI 智能审核（发布时）

```
用户发布商品 → DeepSeekUtils.checkItemPublish(title, description)
    ↓
System Prompt: "你是审核助手，判断物品是否允许上架"
禁止项：违禁品、违法服务、活体动物、处方药、欺诈信息
    ↓
DeepSeek 返回 JSON: {"allow": true/false, "reason": "..."}
    ↓
allow=true → status=1(在售) + 同步ES + 写入PgVector
allow=false → status=4(下架) + 返回拦截原因
AI异常 → status=0(待人工审核)，不阻断发布流程
```

### 4.4 双轨数据同步（MySQL → ES + PgVector）

**第一轨：增量同步（RocketMQ）**

```
商品状态变更 → syncToEs() / deleteFromEs()
    ↓
发送 ItemSyncMessage 到 item-update-topic
    ├── type=1: 上架/更新
    └── type=2: 下架/删除
    ↓
ItemSyncListener 消费消息
    ├── type=1 → doSyncToEs() → itemRepository.save() + ragService.addDocument()
    └── type=2 → doDeleteFromEs() → itemRepository.deleteById()
```

**第二轨：全量同步（管理员触发）**

```
POST /admin/es/sync → syncAllToEs()
    ↓
CompletableFuture.runAsync() 异步执行
    ├── 清除 ES 旧数据 (deleteAll)
    ├── 批量写入新数据 (saveAll)
    └── 逐条写入 PgVector
```

**MQ 失败降级**：catch 异常后直接调用 doSyncToEs() 同步执行，保证数据最终一致。

### 4.5 ES 搜索（search 方法）

```
NativeQuery.builder()
    .withQuery(q -> q.bool(b -> {
        b.must(status=1);                    // 必须在售
        b.must(categoryId, if present);      // 可选分类过滤
        b.must(keyword -> q.bool(should: {   // 关键词搜索
            match(title, keyword),           // IK 分词匹配标题
            match(description, keyword),     // IK 分词匹配描述
            wildcard(title, keyword),        // 通配符（含*或?时）
            wildcard(description, keyword)
        }));
    }))
    .withHighlightQuery(preTags="<span style='color:red'>", postTags="</span>")
    .withPageable(page, size)
    ↓
SearchHits → 提取高亮字段 → 转为 MyBatis-Plus Page<ItemDoc>
```

**IK 分词器**：索引时用 `ik_max_word`（最细粒度切分，提高召回率），搜索时用 `ik_smart`（智能切分，提高精确度）。

### 4.6 搜索建议（suggest 方法）

```java
NativeQuery.builder()
    .withQuery(q -> q.matchPhrasePrefix(m -> m.field("title").query(keyword)))
    .withPageable(PageRequest.of(0, 10))
    ↓
返回前 10 条匹配的 title 列表
```

### 4.7 答辩追问

| 追问 | 应答要点 |
|------|---------|
| 为什么搜索用 ES 而不是 MySQL LIKE？ | MySQL LIKE 无法分词、无法相关性排序、大数据量性能差；ES 支持 IK 分词、高亮、评分排序 |
| ES 和 MySQL 数据一致性怎么保证？ | 双轨策略：增量通过 RocketMQ 异步同步（最终一致性），全量通过管理员手动触发重建；MQ 失败时降级为同步直写 |
| 商品发布为什么要 AI 审核？ | 校园二手平台内容多样，纯人工审核效率低；AI 审核毫秒级响应；AI 异常时不阻断发布，降级为人工审核 |

## 五、AI 智能赋能模块

### 5.1 模块架构

| 层次 | 类名 | 职责 |
|------|------|------|
| Controller | ChatController | AI 文案润色 + 智能导购 |
| Service | RagServiceImpl | 向量存储 + 向量检索 |
| Utils | DeepSeekUtils | AI 审核工具（发布/评价/举报） |

### 5.2 文案润色链路（/ai/polish）

```
前端: 输入标题+描述 → 点击"AI润色"
    ↓
ChatController.polishDescription()
    ↓
System Prompt: "你是二手电商文案写手，生成500字以内精炼文案，不要客套话"
User Prompt: "商品标题: {title}\n原始描述: {description}"
    ↓
chatClient.call(prompt) → DeepSeek 生成润色文案
    ↓
返回文本 → 前端自动填充到描述输入框
```

### 5.3 智能导购链路（/ai/chat）—— RAG 三步流程

```
第一步：向量检索
    ↓
RagService.retrieveRelevantItemDetails(message, topK=3)
    ↓
VectorStore.similaritySearch(
    SearchRequest.query(message)
    .withTopK(3)
    .withSimilarityThreshold(0.0)
)
    ↓
Ollama bge-m3 模型将查询转为 1024 维向量
PgVector 执行余弦相似度搜索
返回匹配商品的 metadata（title, price, description, images）

第二步：上下文拼接
    ↓
systemPromptBuilder:
    ├── 角色定位："你是 SwapU 智能助手"
    ├── 行为约束："简短推荐，不编造商品"
    ├── 商品上下文（如果检索到）：
    │   "1. 标题: {title}, 价格: {price}, 描述: {description}"
    └── 格式约束："友好热情，简明扼要"
    ↓
+ 历史对话（最近6条，即3轮对话，防止上下文过长）

第三步：推理与封装
    ↓
chatClient.call(prompt) → DeepSeek 生成回复
    ↓
返回 { text: AI回复文本, items: 商品卡片数据 }
    ↓
前端同时渲染文字回复 + 可点击商品卡片
```

### 5.4 RagServiceImpl 核心方法

- `addDocument(Item)`：将商品标题+描述拼接为文档内容，metadata 存储 itemId/title/price/description/images，调用 vectorStore.add() 写入 PgVector
- `retrieveRelevantItemDetails(query, topK)`：向量检索，返回包含 metadata 的完整商品信息（供前端渲染卡片）

### 5.5 AI 审核三场景（DeepSeekUtils）

| 场景 | 方法 | 返回格式 |
|------|------|----------|
| 商品发布审核 | `checkItemPublish` | `{"allow": bool, "reason": "..."}` |
| 评价内容审核 | `checkComment` | `{"allow": bool, "reason": "..."}` |
| 举报合理性审核 | `checkReportReasonable` | `{"valid": bool, "reason": "..."}` |

### 5.6 答辩追问

| 追问 | 应答要点 |
|------|---------|
| RAG 的 similarityThreshold 为什么设为 0.0？ | 允许返回所有计算了相似度的结果，由 topK=3 限制数量；阈值过高可能导致检索不到商品 |
| 向量模型为什么选 bge-m3？ | 多语言通用嵌入模型，支持中英文，1024 维精度高；Ollama 本地部署无调用费用 |
| AI 导购为什么不用流式输出？ | 当前用 chatClient.call() 同步阻塞，平均 3.5 秒；后续可切换 chatClient.stream() |
| 如何防止 AI 编造不存在的商品？ | Prompt 硬性指令 + PgVector 实时检索注入真实商品上下文，双重约束 |

### 6.4 物品审核 + 通知联动

```
管理员审核物品 → POST /admin/item/status/{itemId}/{status}
    ↓
更新物品状态
    ↓
同步 ES（status=1 → syncToEs，否则 → deleteFromEs）
    ↓
发送系统通知（@Async 异步）
    ├── 审核通过: "您的物品 [xxx] 已通过审核，现已上架。"
    └── 被下架: "您的物品 [xxx] 已被管理员下架/驳回，请检查是否违规。"
```

**通知类型**：
- type=1：系统通知（审核结果等）
- type=2：交易物流（新订单、支付成功、确认收货等）
- type=3：互动消息

### 6.5 答辩追问

| 追问 | 应答要点 |
|------|---------|
| 管理员权限怎么控制的？ | LoginInterceptor 解析 JWT 获取 role 存入 Request；AdminInterceptor 校验 role==1；两个拦截器在 WebMvcConfig 中注册 |
| 通知为什么要 @Async？ | 通知是辅助功能，不应阻塞主业务流程；异步发送确保主接口响应时间不受通知写入影响 |
| 举报处理涉及什么跨表操作？ | 处理举报时需同步更新物品状态并发送系统通知，涉及 item 表和 sys_notification 表 |
